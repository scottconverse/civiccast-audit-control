## Executive result

The candidate is not ready to advance to the post-audit clean-machine gauntlet as an audited rc17 candidate. Two functional Major defects remain.

First, D4 is still fail-open on supported Setup paths. Complete credentials saved through CivicCast's own provider Setup API are accepted by field-name presence before the real provider settings loader runs. Internet Archive and YouTube can therefore accept a proof reference and report `ready` while the corresponding production adapter still rejects its environment. Subscriber notifications has a second mismatch: readiness validates a webhook secret, but its proof lane maps to the SMTP-backed email provider. Both paths were reproduced at this exact SHA.

Second, the D6 release-matched documentation gives active rc15 testers rc17-only expectations. The changed live guidance explicitly tells users to download rc15, then says that installer automatically provisions Ollama/models and that a resumed setup may request UAC again. Those behaviors were added on this rc17 branch and are not in the named rc15 artifact.

D1, D2, and D3 held under focused execution and source tracing. The D5 copy/reset changes held at source and committed-test level, but actual rerun-Setup/UAC/reboot behavior remains part of the disclosed pristine-machine proof boundary. The exact-artifact clean-machine inference/rehearsal/reboot gauntlet has not run and no class-6 confidence is granted.

## Findings

### CC-RC17-001 - Major - Provider readiness still accepts configuration the real adapters reject

- `civiccast/installer/service.py:468` through `civiccast/installer/service.py:474` returns true when all Setup-stored field names exist, before the real settings-loader branch runs. `record_provider_proof()` relies on that result at `civiccast/installer/service.py:1122`.
- The supported UI-stored YouTube schema contains client ID and client secret at `civiccast/installer/service.py:207`, but the real adapter also requires a refresh token at `civiccast/syndicate/youtube.py:52`. Internet Archive has the same structural bypass: locally stored field names do not make `InternetArchiveSettings.from_env()` resolvable.
- Auditor falsification with all real Internet Archive environment variables absent: save complete Setup credentials, record a proof, then build readiness. The real loader returned its missing-credential error, while Setup moved from `needs_live_proof` to `ready` and the proof response reported `proof_passed`.
- Subscriber notifications checks `CIVICCAST_SUBSCRIBER_WEBHOOK_SECRET` at `civiccast/installer/service.py:369`, while its proof-provider mapping is `email_double_opt_in` at `civiccast/installer/service.py:409`. The production provider constructs SMTP settings at `civiccast/platform/providers.py:135`, whose required inputs are defined at `civiccast/subscribe/smtp.py:45`. With only the webhook secret present, the auditor observed the SMTP loader reject configuration while proof and readiness both reported success.
- The new committed tests cover fictitious and partial environment configuration at `tests/installer/test_provider_readiness_fail_honest.py:40`, but do not cover complete credentials stored through Setup or the webhook-versus-SMTP mismatch.
- **Smallest acceptance criterion:** every provider capable of becoming `ready` must validate the exact credentials consumed by its mapped production adapter. Stored fields may count only if they feed that same adapter path. Add service-level controls for complete stored Internet Archive/YouTube configuration with absent adapter environment, YouTube without a refresh token, and subscriber webhook-only configuration; proof recording must be rejected and readiness must remain non-ready whenever the mapped real adapter rejects configuration.

### CC-RC17-002 - Major - Active rc15 guidance describes rc17-only installer behavior

- `INSTALL-WINDOWS.md:3` and `INSTALL-WINDOWS.md:44` direct testers to the public rc15 installer, while `INSTALL-WINDOWS.md:85` says Setup provisions Ollama and models. That production behavior is introduced only on this branch at `civiccast/apps/installer/src-tauri/resources/headless-bootstrap.ps1:1643`.
- The same release-identity mismatch appears in `FAQ.md:18` and `FAQ.md:28`, `docs/tester/START-HERE.md:3` and `docs/tester/START-HERE.md:66`, and `docs/tester/lpm-beta-test-handoff.md:3` and `docs/tester/lpm-beta-test-handoff.md:79`.
- `docs/tester/known-limitations.md:3` identifies rc15 as current, but `docs/tester/known-limitations.md:41` describes the rc17 UAC-resume behavior as the rc15 target. `INSTALL-WINDOWS.md:98` likewise calls the changed behavior rc15 behavior.
- These are live install and tester-handoff surfaces, so a reader can download the explicitly named rc15 artifact and make installation/storage/UAC decisions from behavior that artifact does not contain.
- **Smallest acceptance criterion:** keep the active rc15 instructions accurate and label new Ollama/UAC behavior as forthcoming rc17 candidate behavior, or update every active release identity only after an exact rc17 artifact is approved and the release-matched handoff is intentionally advanced. Add a documentation consistency control that prevents current-public-release identity from being combined with candidate-only installer capabilities.
