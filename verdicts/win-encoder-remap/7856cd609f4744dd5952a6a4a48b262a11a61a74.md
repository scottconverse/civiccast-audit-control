## Findings

### CC-WER-001 - Major - the informational randomizer leaks into ordinary and required pytest jobs

**Files:** `pyproject.toml:190`, `.github/workflows/deterministic-detectors.yml:97`,
`.github/workflows/ci-test.yml:98`

The slice adds `pytest-randomly>=3.15.0` to the normal `dev` dependency set.
The installed plugin auto-registers with pytest; the auditor's plain
`uv run pytest --trace-config --collect-only ...` reported
`PLUGIN registered: <module 'pytest_randomly' ...>` without `-p randomly`.
No ordinary CI invocation uses `-p no:randomly` or disables plugin autoload.

This contradicts the workflow's stated boundary that `randomized-suite` is the
new informational random-order lane. Required/ordinary `ci-test`, native,
nightly, and local pytest commands now inherit random test ordering whenever
they install the dev group. The exact-SHA required unit job happened to pass,
while the explicitly randomized job and the auditor's plain local full run
failed on different pre-existing order-sensitive tests. That is the flakiness
the informational lane is meant to observe, not inject into every gate.

**Acceptance criterion:** confine `pytest-randomly` to the explicit
informational job, or explicitly disable it in every ordinary/required pytest
lane. Prove with `pytest --trace-config` (or an equivalent executable check)
that an ordinary pytest invocation does not register/reorder through
`pytest_randomly`, while the detector job still does.

### CC-WER-002 - Major - failed randomized runs do not emit a replayable seed

**File:** `.github/workflows/deterministic-detectors.yml:95-101`

The job runs under the Actions shell's fail-fast posture, then enables
`set -o pipefail`. When `uv run pytest ... | tee ...` fails, the shell exits at
that pipeline before executing the subsequent summary/grep commands. In
addition, the command uses `-q`; pytest suppresses
`pytest_randomly.pytest_report_header()`, the only line matching the workflow's
`Using --randomly-seed=...` grep.

The exact failed run `29994106390` contains no replayable random seed. Its
failed test therefore cannot be rerun at the same order, defeating the
detector's core evidence contract.

**Acceptance criterion:** choose or capture the seed before pytest starts,
pass it explicitly, and publish it through an always-run path even when pytest
returns nonzero. Demonstrate one deliberate failed run whose summary/log names
the exact seed and a command that replays that seed.

### CC-WER-003 - Minor - the audit request misidentifies the exact CI failure

The request says the sole randomized-suite failure was
`test_ts_relay.py::test_real_tsp_relay`. Exact run `29994106390`, job
`89163557160`, instead skipped the TSDuck test because `tsp` was unavailable
and failed:

`tests/live/test_real_postgres.py::TestRealPostgresLiveSpineMigration::test_downgrade_drops_live_tables`

The failure was a downgrade guard observing one asset row with
`source_live_session_id`; the run finished `1 failed, 6429 passed, 141 skipped`.
Both the polluting finalization test and downgrade test predate this slice; the
only slice change in that file updates the expected Alembic head. The
auditor's Windows full run did reproduce the separate TSDuck failure in
isolation, and its test/implementation blobs are also identical to base, but
that local result is not the failed GitHub job named in the request.

**Acceptance criterion:** future handoff evidence must quote the actual
exact-SHA CI failure and distinguish it from separate local/base observations.

### 3. Start and reload share pre-flight - proved

Both `GstPlayoutStrategy.start()` and `reload_content()` call
`_resolve_encoder_override()` and pass the returned override into
`graph_from_config()`. The focused reload test passed. Replacing the reload
resolver result with `None` in a disposable exact-SHA mutation worktree made
`test_preflight_reload_content_preserves_software_fallback` fail because the
reload graph no longer contained `openh264enc`.

### 4. H.264 fallback and refusal - proved

With hardware absent and fallback enabled, the built graph contains
`openh264enc` and excludes `mfh264enc`. Without fallback,
`EncoderUnavailableError` is raised and its operator-facing message points to
the channel settings/software fallback. Focused strategy and decision tests
passed.

### 5. Native-Windows HEVC refusal and software x265 exclusion - proved

The software encoder set is checked before the hardware `"265"` refusal.
`libx265` resolves to `x265enc`, is not probed, and proceeds unchanged.
Hardware HEVC is refused even when `mfh265enc` is present. Moving the `"265"`
check above the software set made the named x265 test fail as expected.

### 6. Media Foundation H.264 runtime - proved at confidence class 3

Using
`C:\dev\civiccast-session0-spike\venv\Scripts\python.exe` with
`gstreamer-bundle==1.28.5`, the auditor reran the committed proof:

- `mfh264enc` klass: `Codec/Encoder/Video/Hardware`
- encode: EOS, 5,947,756 bytes, approximately 9.52 Mbps
- decode-back through `tsdemux ! h264parse ! avdec_h264`: EOS
- output SHA-256 prefix: `05af0e8eac270227`

The rerun matched the committed log. The same run reproduced the documented
HEVC boundary: unconstrained `mfh265enc` failed with an internal data-stream
error, while an NV12-pinned chain reached EOS.

### 7. Migration reversibility and old payload compatibility - proved

The focused panel drove real Alembic upgrade-to-head and downgrade-to-0071
against SQLite, observing that 0072 adds and then removes
`allow_software_fallback`. `EgressConfig` retains `extra="forbid"` while the new
field defaults to `False`; an old payload omitting the field validated
successfully. Store/default round-trip tests passed. The exact-SHA Ubuntu
`ci-test` job passed its PostgreSQL-backed ordered suite.
