## Finding

### CC-WS1-007 - Major - Hash-semantics correction remains unguarded and retains one false absolute

- **Dimension:** correctness / documentation / tests / evidence integrity
- **Surface:** `.agent-runs/native-windows/ws1-release-truth/evidence/run_evidence.py:5-10`; `tests/policy/test_release_truth.py`; complete `fafdee3b..f09670ac` diff
- **Evidence:** the diff contains only the runner prose correction and regenerated log. No committed test or executable control asserts `plain named-file == explicit same-path == committed/index blob` and `--no-filters != committed/index blob` for an asserted CRLF working file. The docstring also says that because Git normalizes CRLF, “a disk hash of a Windows checkout never matches” committed content. In this exact Windows detached worktree, the checker file has zero CRLF pairs; its 12,273 disk bytes are byte-for-byte the committed blob content, and both have SHA-256 `da02b6d9c25c316c8aeebd893d3223d3387229c122d4d3c545cc1a935f042487`. `git hash-object --no-filters` likewise equals its committed blob identity `9c9cbb7ed334e394ff676eb833058c98e01923b8`. This directly falsifies “never.”
- **Why it matters:** the corrected primary explanation is currently protected only by prose and an auditor scratch reproduction. The prior false premise survived multiple rounds precisely because no committed control locked the semantics. The remaining absolute also makes the evidence narrative internally inconsistent with the coder’s documented LF-checkout observation.
- **Acceptance criterion:** change “never matches” to an accurate conditional statement, such as raw disk content can diverge when checkout filters change line endings. Add a committed executable CRLF control to the hashed/rerun proof path that asserts named-file and explicit same-path hashes equal the committed/index blob while `--no-filters` differs. Mutation-check that the control fails if the raw form is incorrectly expected to equal the filtered forms, then regenerate `build-verification.log` last. The resulting source change requires a new exact-SHA audit.
