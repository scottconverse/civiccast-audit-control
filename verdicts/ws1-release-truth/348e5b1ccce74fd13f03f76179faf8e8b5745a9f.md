## Findings

### CC-WS1-001 — Major — Superseded-chain corruption can silently pass

- **Surface:** `scripts/policy/check_release_truth.py:74-84`; `docs/releases/release-truth.yaml:47-80`
- **Evidence:** `load_manifest` validates `superseded_by` only when `status == "withdrawn"`. The manifest's rc12-through-rc1 records use `status: superseded` and carry the same relationship. An independent fixture changed a superseded entry's successor to a missing tag; the checker exited 0 and printed `release-truth: PASS`.
- **Consequence:** a typo or deletion can break the rc supersession chain without producing drift, directly falsifying the claim that dangling `superseded_by` values are detected.
- **Acceptance criterion:** validate every supplied `superseded_by` reference and require a valid successor for every status whose contract needs one, including `superseded`; commit a regression in which a superseded entry points to a missing tag and require nonzero drift.

### CC-WS1-002 — Minor — Several cannot-check inputs violate the exit-2 contract

- **Surface:** `scripts/policy/check_release_truth.py:67-93`, `112-116`, `173-197`
- **Evidence:** independent controls supplied (1) a syntactically valid non-mapping YAML document, (2) an invalid historical exemption regex, and (3) a valid JSON object instead of a release list. Each produced an uncaught traceback and process exit 1 rather than the claimed controlled error and exit 2. The committed malformed-YAML test covers parser failure, not these validly parsed but structurally unusable inputs.
- **Consequence:** callers cannot reliably distinguish drift (`1`) from inability to check (`2`), and operator output is an implementation traceback rather than a stable fail-closed diagnostic. The observer still treats either value as non-pass, so this does not create a silent green path.
- **Acceptance criterion:** validate manifest and fixture shapes plus regex compilation at the input boundary; all three controls must emit a concise `ERROR:` message, exit 2, and produce no traceback.

### CC-WS1-003 — Major — The deployed observer's only live run could not perform the release check

- **Surface:** `scripts/policy/check_release_truth.py:96-109`; audit-control workflow `.github/workflows/release-truth-observer.yml`; run 29542136564; audit-control issue #3
- **Evidence:** the checker calls the GitHub releases API without authentication. The first manual observer run at governance SHA `dc03acd` fetched the pinned checker and manifest correctly, but the live fetch returned `HTTP Error 403: rate limit exceeded`. The workflow captured checker exit 2 and opened <https://github.com/scottconverse/civiccast-audit-control/issues/3>. The workflow run itself concluded success because reporting the alarm succeeded.
- **Consequence:** the relay is correctly fail-closed and read-only, but the hosted observer has not yet demonstrated that it can observe release truth. Shared-runner unauthenticated rate limits can turn scheduled checks into recurring availability alarms rather than drift evaluations.
- **Acceptance criterion:** use a rate-limit-resilient read-only retrieval path for the observer, then rerun the actual workflow and show checker exit 0 against live releases and live `main`; close issue #3 through the observer's normal pass path. Release authority must remain absent.

### CC-WS1-004 — Major — Committed build evidence is not bound to the audited source

- **Surface:** `.agent-runs/native-windows/ws1-release-truth/evidence/build-verification.log:1-20`; `CLAUDE.md:100-102`
- **Evidence:** the committed log explicitly records `source SHA at run: 44463a...`, the pre-change base, while the audited implementation is `348e5b1c...`. It contains commands and summarized outputs, but no tested-file hashes or environment record that can bind the uncommitted working bytes used during that run to the committed head. The independent audit reruns prove current behavior for this verdict, but they do not repair the durable coder-evidence provenance required by the repo protocol.
- **Consequence:** a later reader cannot establish from the committed artifact that its green transcript exercised the bytes in the audited commit.
- **Acceptance criterion:** provide immutable final-source proof (for example, a CI run at the new audited commit) or a committed evidence record that cryptographically identifies the tested checker, manifest, tests, and relevant README bytes, and records the required environment, exact commands, outputs, and exits.
