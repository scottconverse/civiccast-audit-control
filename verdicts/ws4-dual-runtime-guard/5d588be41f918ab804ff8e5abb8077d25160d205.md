## Findings

### CC-WS4-001 - Critical - A nonzero WSL inventory command can become confirmed absence and authorize native start

- **Contract:** D3 allows selector-absent native start only after confirmed absence of the CivicCast WSL install. Unknown/error states are non-authorizing and retryable.
- **Code:** `detect_wsl_install()` at `civiccast/native/win_probes.py:442-464` catches invocation exceptions but ignores `CompletedProcess.returncode` and stderr; it returns `False` whenever the distro name is absent from stdout, including a failed command with empty stdout.
- **Executed falsification:** the auditor supplied a runner for which the service query times out and the confirmatory `wsl.exe -l -q` returns exit 1 with `Access is denied`. `detect_wsl_install` returned `False`; `_classify_absence_via_detect` converted A2 to `negative`; `decide()` returned `action=start` for selector absent and otherwise clear probes.
- **Why the exhaustive table does not catch it:** the table correctly treats `None` as non-authorizing, but this boundary bug feeds it a false `False`/`negative` pair instead of the real unknown state.
- **Consequence:** a WSL API/permission failure can be interpreted as proof that no WSL product exists, defeating the guard's primary no-double-runtime premise.
- **Smallest acceptance criterion:** `detect_wsl_install` must return `False` only after a successful, parseable inventory command that positively establishes absence. Every nonzero exit, timeout, invocation error, or untrustworthy output must return `None`. Commit the exact end-to-end expected-red control above and assert `blocked_probe_unavailable`, never `start`.

### CC-WS4-002 - Critical - The supplied WSL maintenance readers fail open on unreadable or malformed authority

- **Contract:** D4 and D7a require every WSL start path to refuse when the maintenance authority is held **or cannot be read**. The request specifically claims fail-closed WSL artifacts.
- **ExecCondition artifact:** `civiccast-guard-execcondition.sh:49-52` discards the registry command status and stderr with `|| true`. At lines 74-87, a missing `Maintenance` parse line simply falls through to `exit 0`. With readable `ActiveRuntime=wsl`, access denied, a failed query, a wrong registry type, malformed JSON, or an unknown maintenance state is therefore authorizing rather than fail-closed.
- **Rust keeper artifact:** `keeper-patch.diff:121-128` returns `false` from `read_maintenance_held()` on every command failure or parse failure and uses substring matching rather than validating the maintenance record. The caller treats `false` as permission to continue.
- **Evidence gap that exposed it:** the recorded seven-case shell panel covered maintenance held/released/absent, but not unreadable or malformed maintenance. `test-notes.md` proposes this future proof rather than carrying an executable control.
- **Consequence:** if routed as supplied, the WSL transmitter can start or remain active during a migration/upgrade freeze when it cannot establish that the interlock is free.
- **Smallest acceptance criterion:** both WSL readers must distinguish a deliberately supported free state from every unreadable, wrong-type, malformed, unknown-state, and command-failure state; only the former may authorize. If `reg.exe` cannot reliably distinguish absent from unreadable, provision a validated released record or use an API/helper that can. Add executable expected-red controls for access denied/nonzero exit, malformed JSON, wrong type, unknown state, held, released, and the chosen absent policy, then exercise the real systemd `ExecCondition` after the owner lands it.

### CC-WS4-003 - Critical - The public cutover command runs outside the required D7a transfer bracket

- **Contract:** `spec-dual-runtime-guard.md` D7a says cutover runs inside the maintenance/freeze interlock. The migration spec requires the owner run ID and generation to remain stable through ownership transfer.
- **Code:** `run_cutover()` at `runtime_cli.py:498-505` explicitly delegates the interlock to an unspecified future caller. The public `cutover-to-native` command at lines 571-580 calls `run_cutover(state_dir=...)` directly with no interlock owner/generation input and no precondition check.
- **Executed falsification:** the auditor patched `runtime_cli.read_interlock` to raise if called, injected harmless phase actions, and invoked `run_cutover`. It completed all five phases with `journal.ok=True`; the asserted interlock read was never reached.
- **Consequence:** the supported operator command can stop/write/switch ownership while a start path or writer is not frozen, defeating the transaction bracket D7a exists to enforce.
- **Smallest acceptance criterion:** the public command must either acquire and hold the D7a interlock for the full transaction or require and continuously verify a caller-owned held record bound to the journal's owner run ID and generation. Free, released, unreadable, wrong-owner, or generation-drift states must exit nonzero before selector mutation. Commit direct-CLI controls for both directions, including tamper during the transaction.

### CC-WS4-004 - Major - GuardMonitor's documented real A3 callable denies its own second evaluation

- **Contract:** D5 continuously re-evaluates every 30 seconds and before every child start. `GuardMonitor` documents that ws5 wires the real Windows-backed callables from `win_probes.py`.
- **Code:** every `GuardMonitor` evaluation calls `self._mutex()` (`runtime_guard.py:403`). `RuntimeOwnerMutex.acquire()` (`win_probes.py:567-613`) always creates/opens and waits again, even when that object already owns the mutex; it has no stable self-owned result.
- **Executed Windows falsification:** with `mutex=RuntimeOwnerMutex(...).acquire` and all other real-contract inputs clear, evaluation 1 returned `start`. Evaluation 2 returned `refuse`, named A3, because this unelevated process could not reopen the production-DACL object it already owned. A direct two-call control reproduced `acquired` then `denied`.
- **Consequence:** the obvious documented production composition starts and then controlled-stops itself within one monitor interval. Under a token that permits recursive acquisition, the same implementation can overwrite the retained handle and recursion ownership instead.
- **Smallest acceptance criterion:** define one lifetime-safe ownership API. Acquisition must happen once and remain held; later monitor/pre-child probes must report stable self-ownership without reopening, overwriting, or recursively acquiring. Add a real-Windows integration control that wires the production object into `GuardMonitor`, performs at least two evaluations plus `pre_child_start`, observes start each time, proves a second process is denied, and proves one shutdown releases the object without handle growth.

### CC-WS4-005 - Major - Cutover phase 2 converts registry failures into successful cleanup

- **Contract:** D7 phase 2 removes keeper Run entries from every loaded hive and enumerates unloaded profiles into evidence. Failure at a phase must remain resumable, not be marked done.
- **Code:** `_default_phase2_remove_run_entries()` (`runtime_cli.py:372-400`) treats every `EnumKey` `OSError` as end-of-list and every per-hive open/query/delete `OSError` as `continue`. `_enumerate_unloaded_profiles()` has the same all-error-as-end/empty behavior. Unlike the corrected A1 scanner, these paths do not distinguish winerror 259 from access denied or mid-scan failure. Fresh phase execution is marked done without invoking the phase verifier.
- **Executed falsification:** forcing `winreg.EnumKey` to raise a PermissionError returned `removed=[]`, `unloaded=[]`, and normal success. No error reached the journal.
- **Consequence:** cutover can write `ActiveRuntime=native` and report success while an unreadable loaded hive still carries the WSL keeper entry, and its unloaded-profile evidence can falsely claim an empty inventory.
- **Smallest acceptance criterion:** only the real end-of-enumeration sentinel may end enumeration. Every other enumeration/open/query/delete failure must fail the phase with hive/index context. Verify the no-marker postcondition on the fresh execution before committing phase 2 done. Add controls for mid-enumeration access denied, per-hive open denial, deletion failure, and unloaded-profile enumeration failure.

### CC-WS4-006 - Major - Rollback returns success when its required keeper restoration did not occur

- **Contract:** D8's final phase restores the keeper for the invoking user; a successful rollback returns the WSL line to service.
- **Code:** `--exe-path` is optional. `_default_phase4_restore_run_entry(None)` at `runtime_cli.py:627-633` returns the normal detail `Run entry NOT restored` rather than failing. `_run_phase` does not call a verifier after a fresh action, so the phase is recorded done and `journal.ok` becomes true.
- **Executed falsification:** with the other rollback phases injected as successful and the production phase-4 action left intact, `run_rollback(exe_path=None)` returned `journal.ok=True`; phase 4's own detail said the Run entry was not restored.
- **Consequence:** the CLI can exit 0 after transferring the selector and enabling WSL while leaving the login keeper absent, so rollback is not durable across logout/reboot.
- **Smallest acceptance criterion:** preflight a valid installed executable path before any rollback mutation, bind it in the journal, make absent/invalid input fail, and verify the exact Run value immediately after writing it before phase 4 can be done. Commit a direct-CLI expected-red for omitted/invalid path and a positive real-registry round trip.

### CC-WS4-007 - Major - Journal and runtime-evidence checks can silently replace or accept the wrong proof

- **Contract:** D7 requires a consumable resumable journal and an evidence file containing timestamps, probe results, removed entries, and unloaded profiles. AC5 requires interruption/resume falsification per phase.
- **Malformed journal:** `_load_journal()` (`runtime_cli.py:160-166`) returns `None` for invalid/unreadable content. Both transactions then create a new run and overwrite the only journal. The auditor placed `{not-json` at the canonical path; cutover silently replaced it and returned `ok=True`.
- **Incomplete/unbound evidence:** `_write_evidence()` runs inside phase 5 before phase 5 is recorded done. The auditor's successful fresh cutover produced evidence containing only phases 1-4 and keys `direction/errors/phases/run_id/unloaded_profiles`; no probe results exist. `_default_phase5_verify_for()` accepts the lexically latest parseable JSON without validating schema, run ID, direction, phase set, or required fields, so a stale unrelated JSON file can satisfy resume.
- **Missing falsification inventory:** the committed CLI suite has one injected phase-2 exception and selected resume-tamper tests; it does not interrupt and recover each D7 phase as AC5 requires, nor test corrupted/truncated journal writes.
- **Consequence:** power loss/corruption can erase the recovery record, and a run can claim complete runtime evidence using a stale or structurally incomplete file.
- **Smallest acceptance criterion:** use atomic durable journal replacement and fail closed while preserving any invalid journal for diagnosis; never silently discard an incomplete opposite-direction run. Bind the exact evidence path and schema to the journal/run ID, include the required probe snapshot and all committed phase outcomes, and validate those bytes rather than any latest JSON. Add kill/failure-and-resume controls at every cutover and rollback boundary plus truncated/corrupt/stale/wrong-run evidence controls.
