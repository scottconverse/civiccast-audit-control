## Findings

### CC-WS5-PKG-001 - Blocker - Open owner decision; coder-owned sampler defects are closed

- **Files:** `.agent-runs/native-windows/ws5-packaging-closure/audit-request.yaml:98-107`;
  `scripts/verify_native_runtime_closure.py:851-1058`, `:1284-1343`,
  `:1601-1654`
- **Evidence:** the former fixed-ceiling scan is gone. The implementation now
  requests `ProcessHandleInformation` for the current process, walks the returned
  snapshot, and requires a real `CreateFileW` canary to appear in sampled paths.
  The auditor reran the above-old-ceiling test and positive-control failure tests;
  all passed. The live D6 run observed 18 open file handles over 106 polls and
  returned 6 PASS / 0 FAIL / 0 SKIPPED.
- **Design judgment:** the fix changes the mechanism, not merely the old value
  16384. The canary makes zero-sample and ineffective-sampler outcomes fail
  closed. There is no realistic success path in the current child runner where
  the sampler contributes no observation: the canary must be present in the
  sampler output before `dynamic_trace` can pass.
- **Limit boundary:** the documented transient-open, single-process, and
  load-then-unload gaps are accurate. `NtQueryInformationProcess` and
  `ProcessHandleInformation` are internal Windows interfaces, but loss or change
  of that interface fails the canary rather than silently passing.
- **Still open:** the authoritative D6 workload versus scope-amendment question
  has not been decided by Scott. Until Scott either requires the full D6 workload
  or approves the narrower scope, the requested charter gate cannot receive
  `AUDIT_PASS`.
- **Required change:** owner action, not an assumed coder change: Scott must
  record either (a) an approved scope amendment, or (b) a decision requiring the
  full D6 workload, after which the coder implements and proves it.

### CC-WS5-PKG-006 - Major - Cleanroom outputs are not bound to one launch or to guest-derived source identity

- **Files:** `.agent-runs/native-windows/ws5-packaging-closure/evidence/cleanroom-gate.sh:55-65`,
  `:82-109`; `.agent-runs/native-windows/ws5-packaging-closure/evidence/cleanroom-closure-build.ps1:58-80`
- **Evidence:** the gate clears one shared `out` directory, writes the expected SHA
  to shared `SOURCE-SHA.txt`, and launches the sandbox without a nonce. The guest
  does not derive the SHA from `C:\repo`; it merely echoes the host-written shared
  file. The host then accepts any log containing the SHA anywhere
  (`if sha not in log`) plus matching artifacts. The guest script itself is copied
  from `$REPO`, not from the detached `$MOUNT`.
- **Consequence:** an overlapping guest left by an interrupted earlier launch can
  read the newly overwritten expected-SHA file while still building its older
  mapped worktree. It then labels the old tree with the new SHA. When the
  packaging outputs are unchanged across those commits, as they are across
  several evidence-only commits in this slice, the host comparison passes.
  The live round-4 cleanroom run passed and the artifacts are byte-identical, but
  that proves this run's output, not that the gate design rejects a stale or
  mismatched producer.
- **Required change:** namespace every cleanroom invocation with an unpredictable
  run ID and unique input/output directory; make the guest echo and the host
  require that exact run ID; bind the guest script to the detached mount; and
  make the guest validate source identity from the mapped source or a
  host-generated manifest tied to that mount, rather than trusting a mutable
  shared label. Parse an exact structured SHA field instead of substring
  membership.

### CC-WS5-PKG-012 - Major - An unledgered or unpinned license text can be staged as authoritative

- **Files:** `civiccast/native/license_texts/__init__.py:255-266`, `:284-327`;
  `scripts/build_native_runtime_closure.py:1040-1085`
- **Evidence:** `available_license_texts()` trusts every `*.txt` found on disk.
  `verify_bundled_license_texts()` iterates only `LICENSE_TEXT_SOURCES`; it does
  not reject an on-disk text absent from that ledger, and it deliberately skips
  every ledger entry with an empty hash. The auditor added `Bogus-1.0.txt` only
  in a temporary redirected license-text directory. Integrity verification
  returned normally, and `write_license_texts(..., ["Bogus-1.0"])` copied it
  into the output as the required notice. The Microsoft pointer is likewise
  unprotected against content drift because its ledger hash is empty.
- **Consequence:** a future classifier can introduce a new license and satisfy
  the build merely by adding a same-named text, with no immutable provenance or
  integrity pin. A modified Microsoft pointer is also blessed into a fresh
  manifest. This defeats the stated design invariant that every shipped notice
  is provenance-pinned before the manifest attests to it.
- **Required change:** require exact set equality between on-disk text IDs and
  `LICENSE_TEXT_SOURCES`, require every selected license to have a ledger entry,
  and apply an explicit integrity policy to every entry. The CivicCast-authored
  Microsoft pointer can use a checked-in expected content hash even though it
  has no upstream URL. Add negative tests for an unledgered extra text and a
  changed pointer.

### CC-WS5-PKG-011 - Minor - Basename comparison is fixed, but invalid basename input can still escape the Boolean contract

- **File:** `scripts/build_native_runtime_closure.py:265-317`
- **Evidence:** full paths and separators are now rejected, and the comparison is
  basename-to-basename. The planted-DLL test passes. However,
  `api_set_resolves_as_a_real_contract("api-ms-win-crt-heap-l1-1-0.dll\0junk")`
  raises `ValueError: embedded null character`; the function catches only
  `OSError`. The current validation rejects three separator characters rather
  than defining the accepted API-set basename grammar.
- **Consequence:** this remains corroboration-only and cannot alter packaging
  acceptance, so the impact is limited. Its advertised `str -> bool` behavior is
  nevertheless not total over string input, and the round-3 fix addresses the
  revealing path case rather than the complete invalid-input class.
- **Required change:** validate against a strict API-set basename pattern (and
  reject empty/NUL/invalid names) before loading, or catch the loader's
  `ValueError` as a false result. Add invalid-name tests including embedded NUL.
