## Findings

### CC-WS5-PKG-001 - Blocker - The required dynamic closure proof does not exist

- **Surface:** `.agent-runs/native-windows/specs/spec-packaging-closure.md:21-28`,
  `:57-69`; `scripts/verify_native_runtime_closure.py:634-679`, `:884-968`;
  `.agent-runs/native-windows/ws5-packaging-closure/audit-request.yaml:71-74`
- **Evidence:** D2(d) requires the D6 suite to run under a file-access trace and
  fail when a non-OS access resolves outside the tree. AC4 and AC5 explicitly depend
  on that trace. The implementation performs a PE import walk, factory lookup,
  plugin backing-file containment check, one caption round trip, and GPL checks. It
  does not trace `LoadLibrary`, delay-time behavior at feature use, data/resource
  opens, or all representative application paths. The request admits the trace is
  absent.
- **Consequence:** a package can pass while a factory or support library opens an
  unshipped DLL/data file at runtime. The current D6 pass proves one caption path and
  plugin registry containment, not the authoritative D2/D6 closure contract. A known
  gap is not a waiver of an acceptance criterion.
- **Gate effect:** blocks `charter-section-7-runtime-packaging`.
- **Acceptance criterion:** implement the D2(d) trace in
  `scripts/verify_native_runtime_closure.py` or a checked-in companion invoked by it;
  record every file opened by the representative D6 paths; classify OS accesses
  explicitly; fail on every non-OS path outside the tree; add a real outside-tree
  runtime-load negative control and a non-PE resource deletion control that fail for
  the traced access, not only because the manifest notices deletion. Re-run the
  spec's representative application-path suite and commit exact-SHA-bound trace
  artifacts.

### CC-WS5-PKG-002 - Critical - Both GPL distribution gates can be bypassed by a valid project-name variant

- **Surface:** `civiccast/native/runtime_closure.py:36-49`, `:248-310`;
  `scripts/build_native_runtime_closure.py:240-268`, `:815-835`;
  `tests/native/test_runtime_closure.py:91-169`
- **Evidence:** both gates compare raw distribution strings to an underscore/lowercase
  denylist. This exact counterexample passed both gates:

  ```text
  assert_no_gpl_distributions(["GStreamer_Plugins_GPL"])
      -> returned normally

  select_plugin_seeds(
      {"appsrc": (".../gstapp.dll", "GStreamer_Plugins_GPL")},
      required={"appsrc"},
  )
      -> returned the plugin
  ```

  Python distribution names are case-insensitive and normalize runs of `-`, `_`, and
  `.`. The gate does neither. It also denies only two known names instead of requiring
  the staged set to equal the six authorized runtime inputs.
- **Consequence:** the asserted "refuses if a GPL distribution is staged at all"
  invariant is false. A renamed/case-variant distribution can cross the input boundary,
  and a non-excluded factory attributed to it can cross the selection boundary.
- **Gate effect:** blocks the no-GPL gate.
- **Acceptance criterion:** in `runtime_closure.py` and the RECORD/dist-info parsing
  path, canonicalize project names using the packaging/PEP 503 rule
  (`re.sub(r"[-_.]+", "-", name).lower()` or the standard library used by the
  project); enforce an allowlist of the exact six permitted runtime distributions,
  not only a two-name GPL denylist; apply the same canonical identity in seed
  selection; and add negative tests for case, hyphen, underscore, dot, and an
  unexpected staged distribution at both gates.

### CC-WS5-PKG-003 - Critical - Prefix allowlists deliberately fail open on nonexistent OS and host-runtime DLLs

- **Surface:** `scripts/build_native_runtime_closure.py:138-193`, `:342-355`
- **Evidence:** `is_system_dll("api-ms-win-civiccast-fake-l99-99-99.dll")` and
  `is_system_dll("ext-ms-win-civiccast-fake-l99-99-99.dll")` both returned `True`.
  `HOST_RUNTIME_DLL_PREFIXES = ("python3",)` similarly accepts `python313.dll` or
  any other `python3*` import while the recorded requirement remains CPython 3.12.
  Imports matching these prefixes are removed before `resolve_pe_closure` can reject
  them. All 40 exact DLLs currently listed were present in both System32 and SysWOW64
  on the audit host, but that does not make arbitrary API-set names real or compatible
  with Windows 10 1809.
- **Consequence:** a future or tampered PE can name a nonexistent API-set or a
  different CPython minor and still produce a "complete" tree. The code comment says
  a CPython bump should not become unresolved; that is precisely the drift the closure
  must refuse until the installer/runtime contract changes with it.
- **Gate effect:** blocks the fail-loud closure and OS/host dependency claims.
- **Acceptance criterion:** replace the unbounded prefix exemptions with a
  checked-in, reviewed inventory derived from the exact pinned PE set; validate every
  API-set contract against the declared minimum Windows build; allow exactly
  `python312.dll` for the current `cp312` extensions (or derive and cross-check both
  values); add fabricated API-set and `python313.dll` negative controls that must
  raise `UnknownProvenanceError`.

### CC-WS5-PKG-004 - Major - Required license texts are not bundled

- **Surface:** `.agent-runs/native-windows/specs/spec-packaging-closure.md:30-37`;
  `scripts/build_native_runtime_closure.py:506-784`;
  `.agent-runs/native-windows/ws5-packaging-closure/audit-request.yaml:89-93`
- **Evidence:** D3 says required notices are bundled. The builder writes summaries
  copied from wheel METADATA, and its generated README explicitly says the full
  upstream license texts are not available and must be sourced separately. The
  audit request lists this as an unsolved limitation.
- **Consequence:** the tree is described as the runtime the installer will carry,
  but it omits the actual LGPL/MPL/FTL/etc. texts needed for compliant redistribution.
  A limitation is not a fulfilled packaging requirement.
- **Gate effect:** blocks this packaging gate.
- **Acceptance criterion:** add version-pinned upstream license/notice texts for every
  shipped component under `licenses/`, record source URL and content hash for each,
  include those files in `runtime-manifest.json`, `SHA256SUMS`, and `LICENSE-BOM.md`,
  and add a negative control proving a missing or altered required text makes the
  build/verification fail.

### CC-WS5-PKG-005 - Major - FreeType is a fourth, unreported dual-license election

- **Surface:** `civiccast/native/runtime_licenses.py:197-245`, `:288-296`;
  `.agent-runs/native-windows/ws5-packaging-closure/evidence/license-provenance.md:471-491`
- **Evidence:** the BOM elects `FTL` for `bin/freetype-6.dll` and
  `lib/girepository-1.0/freetype2-2.0.typelib`. The evidence memo says the only
  dual-license elections are librtmp, cairo, and D-Bus. FreeType's own official
  license page says FreeType offers a choice between FTL and GPLv2.
- **Consequence:** the resulting non-GPL classification is supportable, but the
  owner-decision inventory is incomplete in exactly the category the request asked
  the auditor to exhaust.
- **Gate effect:** blocks the license-audit claim until disclosed; it does not prove
  GPL bytes are present.
- **Acceptance criterion:** update `runtime_licenses.py`, `license-provenance.md`,
  the generated BOM/notice source, and the audit request's owner-decision register to
  identify FreeType's `FTL OR GPL-2.0-only` choice and the elected FTL branch; add a
  test that the complete dual-license election set is explicit and cannot silently
  lose an entry.

### CC-WS5-PKG-007 - Major - The Windows 10 1809 floor is asserted, not derived

- **Surface:** `scripts/build_native_runtime_closure.py:138-152`
- **Evidence:** Microsoft documents the base `ID3D12Device` interface as introduced
  in Windows 10. That establishes a Windows 10 family floor, not specifically build
  17763/1809. The code supplies no per-import minimum-version ledger or test against
  1809 and accepts all API-set prefixes without version validation.
- **Consequence:** "Windows 10 1809; d3d12.dll is the binding constraint" conflates
  the chosen product support floor with a dependency-derived minimum. An installer
  could rely on this unsupported statement when deciding whether to install.
- **Gate effect:** blocks the claimed OS dependency proof.
- **Acceptance criterion:** either relabel 1809 as the owner-selected product policy
  floor and separately state the dependency-derived floor, or provide authoritative
  minimum-client evidence for every exact and API-set import proving 17763 is the
  binding build. Add a checked test/ledger that fails when a new allowlisted import
  raises or invalidates the floor.
