## Findings

### CC-WS5-PKG-016 - Minor - The structured cleanroom identity accepts a false `file_count`

- **Files:** `.agent-runs/native-windows/ws5-packaging-closure/evidence/cleanroom_identity.py:40`,
  `:85-86`, `:96-132`, `:147`
- **Evidence:** the auditor constructed otherwise valid identity records whose
  nonce, complete-tree fingerprint, and three artifact hashes matched the
  expected host/guest values. A truthful `file_count=3246`, a false
  `file_count=1`, and JSON `file_count=true` were all parsed and accepted with
  an empty error list. `bool` passes the current `isinstance(value, int)`
  predicate, and `verify()` never compares the field to the independently
  computed mount count.
- **What did resist attack:** half-written JSON, a valid record followed by a
  NUL, and two concatenated valid records were rejected. Wrong nonce,
  fingerprint, and artifact bytes are checked by equality. The Python and
  PowerShell complete-tree fingerprint implementations independently produced
  the same digest and count:
  `a77768190c855bcbc41f3b6b6e6b9c418619206592fdc91dd95af17304999bae`,
  3246 files, on the post-gate exact-SHA mount.
- **Consequence:** this does not provide a stale guest with the current
  mapping, and the matching fingerprint still binds the source bytes.
  Nevertheless, `identity.json` is described and accepted as one exact,
  structured, guest-derived identity record while one of its required fields
  can be false. That is an evidence-integrity false pass in the mechanism
  specifically redesigned to reject records that parse but lie.
- **Smallest acceptance criterion:** either remove `file_count` from the
  accepted schema, or require `type(file_count) is int` and compare it with the
  independently computed host mount count. Add controls proving `true`, zero,
  the wrong positive count, and a correct count are respectively rejected,
  rejected, rejected, and accepted through the real CLI verification path.
