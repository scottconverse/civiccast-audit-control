## Findings

### CC-WS5-PKG-013 - Blocker - The required exact-SHA gate is red on a real inaccessible open-file path

- **Files:** `scripts/verify_native_runtime_closure.py:974-998`,
  `scripts/verify_native_runtime_closure.py:1000-1009`,
  `tests/native/test_closure_verifier.py:626-694`
- **Evidence:** the exact-SHA gate reached the native suite and returned
  `2 failed, 728 passed`. Both real handle-enumeration tests raised
  `PermissionError: [WinError 5] Access is denied` at
  `Path(path).is_dir()` for
  `C:\$Extend\$Deleted\00030000002A32BC0B0A7461`. Five independent focused
  reruns reproduced both failures every time (10 further failed test
  invocations). This is not a stale coder log or a one-off assertion mismatch.
- **Mechanism:** `_own_open_file_paths()` resolves every disk handle to a name,
  then performs a pathname `stat()` via `Path.is_dir()`. Windows can return a
  valid handle path under the NTFS deleted-object namespace that the process
  cannot stat by name. The background sampler catches the exception around the
  whole poll and discards every already-observed path from that poll; the direct
  tests propagate it. The D6 child happened not to inherit this handle and
  passed, which makes the gate environment-dependent rather than closing the
  defect.
- **Consequence / blast radius:** the named exact-SHA gate cannot pass on the
  actual audit host, and the sampler can discard complete polls because one
  unrelated handle is unstatable. That directly affects the packaging gate's
  native-access evidence and prevents advancement.
- **Required change:** classify directory-vs-file from the handle without a
  pathname `stat`, or handle an unstatable handle per-entry without discarding
  the rest of the poll. Add a deterministic regression for an inaccessible or
  deleted-object handle, prove the native canary and above-16384 handle remain
  observable in the same poll, then rerun the full exact-SHA gate cleanly.

### CC-WS5-PKG-006 - Major - Still open: the cleanroom identity binding is partial and replayable

- **Files:** `.agent-runs/native-windows/ws5-packaging-closure/evidence/cleanroom-gate.sh:42-77`,
  `:142-176`;
  `.agent-runs/native-windows/ws5-packaging-closure/evidence/cleanroom-closure-build.ps1:65-111`;
  `.agent-runs/native-windows/ws5-packaging-closure/evidence/source_fingerprint.py:13-30`
- **Evidence:** a detached stale mount at
  `7c5beb6c754278b38e795f2148246974b781a243` and the audited HEAD produced the
  identical guest-derived fingerprint
  `d3c25901ad8604ea6bb16719b93e71b509305f0727eb409ae894c0570614dd1f`
  over 138 files. That is expected from the implementation: it hashes only
  `civiccast/native`, `scripts`, and `requirements-native-runtime.txt`, not the
  exact tracked tree or commit. The committed cleanroom log itself is bound to
  `7c5beb6c`, not the audited HEAD.
- **Replay result:** the guest still reads the SHA and nonce from mutable,
  shared host files. The host accepts identity by substring membership
  (`if sha not in log`, and the same for nonce/fingerprint), not exact structured
  fields. Starting from the committed `7c5beb6c` log, appending the current SHA,
  a new nonce, and the expected fingerprint made all three identity predicates
  true while the old SHA remained in the log. The three artifact hashes also
  remain identical across `7c5beb6c` and current HEAD.
- **Additional binding gap:** `cleanroom-gate.sh` still copies the guest script
  from `$REPO`, not the detached `$MOUNT`, so the executable guest harness is
  not itself bound to the mounted SHA. The stable shared input directory also
  lets an older mapped guest read a newly written nonce; the nonce is not a
  capability unique to one mapping.
- **Consequence / blast radius:** an old guest/source mapping with unchanged
  packaging roots can be represented as the current exact SHA, and a copied log
  can satisfy the identity parser by retaining contradictory values and
  appending expected substrings. The fresh `7c5beb6c` run is useful evidence for
  that commit's packaging bytes, but it does not close the round-4 exact-source
  and exact-launch acceptance criterion.
- **Required change:** use a unique per-launch mapped shared directory that a
  stale guest cannot discover through a stable path; bind the guest script to
  `$MOUNT`; derive or verify a fingerprint over the complete tracked source
  state required for exact-SHA identity; and parse one exact structured record,
  rejecting duplicates, extra identities, wrong SHA, replayed nonce, wrong
  fingerprint, and copied/tampered logs. Add executable negative controls for
  each case.

### CC-WS5-PKG-014 - Major - The HEVC retraction did not reach the executable dependency declaration or its test

- **Files:** `.agent-runs/native-windows/ws5-packaging-closure/evidence/hevc-store-extension-finding.md:3-60`;
  `scripts/verify_native_runtime_closure.py:1537-1551`;
  `tests/native/test_closure_verifier.py:768-803`
- **Evidence:** the retraction correctly says a loaded Store DLL is not the
  encoder used, the NVIDIA driver MFT performs the HEVC encode, and no packaging
  consequence follows. The executable declaration still states the opposite:
  absence of `Microsoft.HEVCVideoExtension` makes HEVC hardware encoding
  unavailable, the installer leaves a gap, and an owner decision is owed. Its
  test repeats that Media Foundation “loads HEVC encoding from” the Store
  package and requires the false `OWNER DECISION OWED` consequence.
- **Consequence / blast radius:** the committed evidence warns readers not to
  act on a finding while the verifier's executable source and regression test
  preserve and enforce the retracted conclusion. A D6 trace that encounters the
  Store package can therefore emit owner-facing packaging guidance the coder
  has already proved false. This undermines the request's claim that the
  disclosure was corrected rather than merely overlaid with prose.
- **Required change:** keep any declaration needed to account for harmless
  enumeration loads, but change its consequence and test to the proved
  “installed/loaded, not necessarily used” semantics. Remove the false
  Store-extension requirement and owner-decision claim, and reconcile the
  retraction's broad hardware statement with its explicit unmeasured AMD-iGPU
  boundary.

### CC-WS5-PKG-011 - Minor - Still open: malformed API-set suffixes pass as genuine contracts

- **File:** `scripts/build_native_runtime_closure.py:265-323`
- **Evidence:** NUL, empty, separator-bearing, colon-bearing, whitespace, and
  40,000-character names now return `False`; mixed-case real input returns
  `True`. However,
  `api-ms-win-crt-heap-l1-1-0` and
  `api-ms-win-crt-heap-l1-1-0.exe` both return `True` on the audit host. The
  loader forwards those malformed names, and the helper equates “forwarded to a
  different basename” with “valid API-set grammar.”
- **Consequence:** this helper is corroboration-only and cannot add to the
  packaging allowlist, so it does not independently compromise the closure.
  It still fails the round-4 requirement to validate the complete basename
  grammar and can produce false live corroboration.
- **Required change:** enforce the accepted API-set basename grammar (including
  required prefix, level/version fields, and `.dll` suffix) before calling the
  loader, then add missing-suffix, wrong-suffix, NUL, empty, separator, overlong,
  mixed-case, and valid-name cases.
