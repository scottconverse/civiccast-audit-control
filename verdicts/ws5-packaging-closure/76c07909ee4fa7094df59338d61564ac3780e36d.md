## Findings

### CC-WS5-PKG-001 remains open - Blocker - The dynamic closure check can pass outside-tree dependencies and does not implement the authoritative D2(d)/D6 proof

- **Files:** `scripts/verify_native_runtime_closure.py:623-765`,
  `:1038-1134`, `:1137-1264`, `:1376-1418`;
  `.agent-runs/native-windows/specs/spec-packaging-closure.md:21-28`,
  `:51-69`
- **Reproduced fail-open behavior:**
  - A real existing file outside the tree under the global temp directory was
    fed as a mapped module. `_dynamic_trace_result` returned `PASS`.
  - A real existing repo file under a `sys.path` root was fed as a mapped
    module. It returned `PASS`.
  - A real existing file under the audit venv / `sys.prefix` was fed as a
    mapped module. It returned `PASS`.
  - A fabricated outside file named
    `Evil.Microsoft.HEVCVideoExtension.Clone.dll` returned `PASS` because
    `_match_declared_dependency` performs an unanchored substring comparison.
    A second undeclared outside file correctly returned `FAIL`. This proves the
    test's “undeclared WindowsApps fails” example but falsifies the broader
    property.
- **Structural gap:** the permitted roots include all of `%TEMP%`,
  `sys.prefix`, `sys.base_prefix`, and every `sys.path` root. They are broad
  allowlists, not exact harness identities. A missing product dependency that
  falls back to the developer venv, repo, or a temp sibling is counted as
  inside a permitted root.
- **Coverage gap:** the CPython hook cannot see the native file-access surface
  of this tree. GStreamer, Media Foundation, fontconfig, GIO, codecs, and GPU
  drivers are native code. Their `CreateFileW` resource/config/cache accesses
  are exactly the behavior D2(d)'s file-access trace was introduced to
  observe. The end-of-run module snapshot also cannot prove a module was never
  loaded and unloaded during registry/factory work.
- **D6 gap:** repository search confirmed that this verifier does not run the
  spec's `SWAPS=2` engine path, D4 one-second encoder probe matrix, packaged
  PostgreSQL `initdb/start/SELECT 1`, NATS JetStream acknowledgement, or
  control-plane import smoke. The new trace therefore wraps only the existing
  factory lookup and caption leg, not the authoritative representative D6
  application paths required by AC2.
- **Current-tree nuance:** the fresh run showed all 33 plugin DLLs still mapped
  at the end, so no plugin DLL itself was demonstrated to unload mid-run on
  this host. GStreamer's helper scanner was absent and it logged “Couldn't
  create helper process,” so a scanner child-process blind spot was not
  material for this exact tree. Those facts do not cure the reproduced broad
  root bypass or native file-access blind spot.
- **Consequence:** an incomplete or contaminated beta artifact can report
  `dynamic_trace PASS`. The check is not sufficient for a beta packaging
  closure.
- **Gate effect:** blocks `charter-section-7-runtime-packaging`.
- **Required change:** in `scripts/verify_native_runtime_closure.py`, replace
  global temp / prefix / import-path root exemptions with exact per-run paths
  and pre-product harness file identities; classify OS modules through the
  reviewed OS dependency inventory rather than permitting all of
  `%SystemRoot%`; match declared Store dependencies by exact structured package
  family/publisher/root and expected module names, not substrings; and use a
  trace capable of observing native opens, load/unload, and spawned processes
  for every authoritative D6 path. If the first-half slice is intentionally
  narrower than D6 in the checked-in authoritative spec, obtain and commit an
  owner-approved spec amendment rather than silently narrowing the verifier.

### CC-WS5-PKG-002 remains open - Critical - The two GPL gates still accept unauthorized distribution identities

- **Files:** `civiccast/native/runtime_closure.py:53-83`, `:301-367`;
  `scripts/build_native_runtime_closure.py:985-1005`
- **Proof that passed:** case, hyphen, underscore, dot, repeated-separator, and
  surrounding-whitespace forms of `gstreamer-plugins-gpl` were refused by both
  `assert_no_gpl_distributions` and `select_plugin_seeds`.
- **Counterexample:** `assert_no_gpl_distributions(["gstreamer-libs",
  "civiccast-unknown-runtime"])` returned normally, and
  `select_plugin_seeds({"appsrc": ("lib/gstapp.dll",
  "civiccast-unknown-runtime")}, required={"appsrc"})` selected the plugin.
- **Consequence:** round 1's accepted exact-input allowlist was not implemented.
  A renamed, replacement, or unexpected distribution can contribute a
  known-looking plugin basename and bypass both distribution gates. The
  per-file classifier then reasons mainly from the basename, so this is not
  harmless staging noise.
- **Gate effect:** blocks the no-GPL / authenticated-input gate.
- **Required change:** define canonical exact sets for (a) the seven staged
  lock identities, including `setuptools` as the non-shipping transitive build
  input, and (b) the six distributions authorized to contribute runtime files.
  Refuse missing, extra, duplicate-claiming, or unauthorized canonical
  identities before selection, and refuse an origin outside the shipping set
  in `select_plugin_seeds`. Add both-gate tests for an unexpected identity and
  a distribution whose `RECORD` claims a known plugin path.

### CC-WS5-PKG-003 remains open - Critical - API-set acceptance is build-host dependent and validates loadability, not OS provenance or target compatibility

- **File:** `scripts/build_native_runtime_closure.py:172-234`
- **Proof that passed:** fabricated API-set names returned `False`; the real
  `api-ms-win-crt-heap-l1-1-0.dll` returned `True`; fabricated/different-minor
  Python names returned `False`; `python312.dll` and `python3.dll` returned
  `True`.
- **Counterexample:** the auditor copied `z-1.dll` to a temporary file named
  `api-ms-win-civiccast-fake-l99-99-99.dll`, added that directory with
  `os.add_dll_directory`, cleared the function cache, and called
  `is_system_dll`. It changed from `False` to `True`.
- **Target gap:** the fresh payload imports 17 API-set contracts. The
  implementation asks the Windows 11 build-26200 loader about them. It does not
  prove those contracts and imported symbols exist on the supported Windows
  10 1809 target. A newer host can accept a target-incompatible contract; an
  older build host can reject an otherwise valid target. This also makes the
  closure decision depend on host state.
- **Consequence:** the prior arbitrary-prefix bypass is narrowed but the claim
  “Windows provides this target dependency” remains unproved and can be
  spoofed by loader search state.
- **Gate effect:** blocks the fail-loud OS dependency and supported-floor
  claims.
- **Required change:** derive and check in the exact API-set contract inventory
  from the pinned PE set; validate it against an authoritative Windows 10 1809
  API-set/schema or an actual clean 1809 target; fail when a new contract
  appears without that review. If a live-loader corroboration remains, verify
  that resolution is an OS API-set contract rather than an arbitrary DLL found
  through process search directories. Keep the exact CPython set.

### CC-WS5-PKG-009 - Critical - `SHA256SUMS` and `LICENSE-BOM.md` can be missing or tampered while the verifier reports 6 PASS

- **Files:** `scripts/build_native_runtime_closure.py:1054-1064`;
  `scripts/verify_native_runtime_closure.py:377-472`
- **Reproduced controls:**
  - Deleted `SHA256SUMS` from a copied fresh tree. Verification returned
    `6 PASS, 0 FAIL, 0 SKIPPED`, exit 0.
  - Appended `AUDITOR TAMPER` to `LICENSE-BOM.md`. Verification again returned
    `6 PASS, 0 FAIL, 0 SKIPPED`, exit 0.
- **Cause:** all three top-level trust artifacts are explicitly excluded from
  the disk cross-check because the builder hashes the payload before creating
  them.
- **Consequence:** AC7's “every shipped file: hash + BOM entry + license,
  cross-checked counts” is false, and the exact artifacts an operator/reviewer
  uses as the trust record are outside the current integrity check.
- **Gate effect:** blocks manifest trust and incomplete/tampered-tree closure.
- **Required change:** generate the payload `SHA256SUMS` and `LICENSE-BOM.md`
  before the final manifest, include both files and their hashes in
  `runtime-manifest.json`, and make verification require and validate them.
  The final manifest itself remains the installer-signature trust root and
  need not hash itself. Add deletion and alteration negative controls for both
  sibling artifacts.

## HEVC external-dependency disposition

The fact that `Microsoft.HEVCVideoExtension` is external does **not**, by
itself, require the entire first-half packaging gate to fail. The authoritative
spec's halt trigger explicitly permits an unshippable media dependency to be
documented as an owner-visible external dependency. “Declare and escalate to
the owner” is therefore the right disposition for the dependency itself,
provided:

1. the declaration is exact and cannot absorb unrelated paths;
2. the beta capability/installer does not claim HEVC works on a base Windows
   installation;
3. clean-machine proof, installer/doctor detection, and an error naming the
   missing extension exist before HEVC support is claimed; and
4. Scott decides whether HEVC is supported or intentionally unavailable on
   stations lacking the Store package.

The current gate is blocked because the declaration matcher and surrounding
trace fail open, not merely because the HEVC package is external. Until the
owner decision, the HEVC capability claim is blocked even if the broader
packaging work later passes.
