## Findings

### CC-WS2-001 - Critical - Equivalence can still pass after semantic DDL or role/permission drift

- **Dimension:** correctness / data integrity / required verification
- **DDL evidence:** `civiccast/dr/restore_drill.py:475-479` reverse-maps `__dr_canon` and the disposable index name with unrestricted `str.replace` calls over the whole `pg_get_indexdef` result. That rewrites SQL string literals and identifiers in predicates, not only the index header. The auditor drove the production `_def_lists_mismatch` with source predicate literal `'__dr_canon'` and restored predicate literal `'civiccast'`, while a fake canon engine returned the corresponding valid Postgres deparse shapes. The exact production transformation returned `None`: `source_literal=__dr_canon`, `restored_literal=civiccast`, `production_diff=None`. Those predicates are semantically different but become byte-identical after the global replacement.
- **Role-discovery evidence:** `civiccast/dr/restore_drill.py:286-306` seeds relevant roles only from table owners and non-PUBLIC table grantees. The report later compares database, schema, table, and sequence owner names (`:1176-1203`), but a role that exclusively owns the database, schema, or a sequence is never included in the attribute/membership check. `civiccast/dr/restore_drill.py:849` retains a `pg_auth_members` edge only when both member and granted role are already in that seed. Its own docstring's example, an app role inheriting `ops_admin`, is ignored whenever `ops_admin` owns/grants no object directly.
- **Executed membership false-pass:** the auditor modeled a source catalog edge `app_owner -> ops_admin`, with `app_owner` in the production seed and `ops_admin` only the membership parent. The unmodified production `_pg_role_memberships` returned `[]` for both source and a standby with the edge absent. The real `run_postgres_cold_standby_drill`, with all unrelated catalog/subprocess surfaces held equal, returned `report_errors=[]` and `report_ok=True`.
- **Privilege evidence:** the only privilege collector is `_pg_table_grants` at `civiccast/dr/restore_drill.py:256-283`. There is no comparison for database CONNECT/TEMP, schema USAGE/CREATE, sequence USAGE/SELECT/UPDATE, or default ACLs. Losing those can leave owner names and table grants equal while the station role cannot connect, resolve the schema, advance sequences, or receive correct privileges on future objects.
- **Control-path evidence:** `tests/dr/test_postgres_restore.py:938-989` changes a role attribute but asserts only that `_pg_role_attributes` differs. Lines `992-1067` revoke a table grant but assert only that `_pg_table_grants` differs. Neither invokes `run_postgres_cold_standby_drill` after mutation, so deleting the corresponding production report comparison would not make those tests red. There is no membership-drift control. The missing-role control at `:870-935` does drive the real report and is valid, but it proves only that narrower case.
- **Consequence:** a drill can report `ok=True` although an index predicate changed meaning, an inherited role relationship disappeared, an exclusive owner role has wrong attributes, or non-table ACLs no longer permit the restored station to operate. That directly undermines the charter's introspected roles/permissions and source/restored equivalence gate.
- **Smallest acceptance criterion:**
  1. Compare server-canonical DDL before any operator-facing presentation mapping, or rewrite only parser-known index-name/schema tokens. Never globally replace arbitrary SQL text. Add a real-Postgres expected-red case where predicate literals `__dr_canon` and `civiccast` remain distinct.
  2. Seed role verification from database/schema/table/sequence owners plus every non-PUBLIC grantee across database, schema, table, sequence, and default ACLs; traverse the relevant `pg_auth_members` graph so every compared edge endpoint and inherited role is included. Compare those ACL classes source versus standby.
  3. Make wrong role attributes, missing/changed membership, revoked table grant, and revoked database/schema/sequence privilege fail `run_postgres_cold_standby_drill` itself (`report.ok is False` with the expected error), not only a helper assertion.
- **Blast radius:** disaster recovery, cold standby, database authorization, schema resolution, sequence-backed inserts, constraint/index semantics, and later native cutover.
