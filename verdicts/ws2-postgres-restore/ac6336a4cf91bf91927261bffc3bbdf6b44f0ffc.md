## Finding

### CC-WS2-001 - Critical - Role and table-ACL comparison still has symmetric false-pass paths

- **Dimension:** correctness / authorization / data integrity / required verification
- **Unexpected-membership path:** `civiccast/dr/restore_drill.py:1412-1414` computes the relevant role set only from the source. Lines `1449-1452` then query standby attributes and memberships using that same source set. `_pg_role_memberships` at `:1189-1209` retains an edge only when **both** endpoints are already in the supplied set. A standby-only role newly granted to, or inherited by, a source application role is excluded rather than reported as unexpected.
- **Executed report-level false-pass:** the auditor held all other source/restored catalog values equal, gave the source no membership edges, and gave the standby catalog the edge `app_owner -> ops_admin`, with `app_owner` in the source-relevant set and `ops_admin` present only on standby. The unmodified production `_pg_role_memberships` returned an empty set on both sides. The real `run_postgres_cold_standby_drill` comparison completed with `report_ok=True` and `report_errors=[]`. This is the symmetric counterpart of the missing-source-edge case fixed in round 4.
- **Incomplete table ACL:** `_pg_table_grants` at `civiccast/dr/restore_drill.py:257-283` selects only `grantee`, `table_name`, and `privilege_type` from `information_schema.role_table_grants`. PostgreSQL 17's authoritative documentation states that `role_table_grants` omits access supplied through `PUBLIC`, and the same view exposes `grantor` and `is_grantable`, neither of which is selected. Thus `GRANT SELECT ... TO PUBLIC` can be invisible, and otherwise identical rows with versus without grant option compare equal. The database/schema/sequence/default collectors correctly retain `(grantee, grantor, privilege, is_grantable)`; the table collector is the outlier. See <https://www.postgresql.org/docs/17/infoschema-role-table-grants.html>.
- **Consequence:** a cold-standby report can say `ok=True` although a relevant station role gained an unexpected inherited privilege, a table became PUBLIC-readable, or a grantee gained/lost the ability to delegate access. That contradicts the charter's introspected source/restored roles/permissions equivalence proof and can hide privilege escalation or unusable restored authorization state.
- **Smallest acceptance criterion:**
  1. Make relevant role/membership comparison symmetric. At minimum, close the standby membership graph from the source-relevant seed and compare the union, attributes, and all touching edges, so any standby-only endpoint connected to a relevant role is included and reported. Add a real-report expected-red control by appending an extra role and `GRANT` edge to the replayed globals artifact; `report.ok` must be false.
  2. Compare complete table ACL entries, including PUBLIC, grantor, privilege, and grantability (for example, with a catalog/`aclexplode` collector consistent with the other object classes). Add real-report expected-red controls for a PUBLIC table grant and a `WITH GRANT OPTION` difference.
- **Blast radius:** disaster recovery, cold standby, database authorization, least privilege, public data exposure, operator access after restore, and later native cutover.
