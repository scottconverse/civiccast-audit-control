## Scope and claim results

The source contains a real Postgres helper chain: custom-format `pg_dump`, drop/create of a fresh database, `pg_restore` over stdin with fail-fast behavior, restored-table snapshots, migration-revision checks, extension-name checks, sequence-name checks, and app-store read-through. The two named negative controls call the production comparison and restore functions, not reimplementations. CI run `29559102912` is bound to the audited head and executed all three tests without skips.

Those facts do not satisfy the requested charter gate. The charter's verification depth requires source/restored equivalence for application tables, sequences, constraints, extensions, roles/permissions, and data. The current report can pass while those states differ. The exact shipping `run_full_drill`/CLI/installer Postgres path also remains unexecuted: CI manually assembles a manifest and calls lower-level helpers with test-only Docker command prefixes.

## Findings

### CC-WS2-001 - Critical - The equivalence report can pass with materially different restored database state

- **Dimension:** correctness / data integrity / required verification
- **Evidence:** `civiccast/dr/restore_drill.py:40-69` iterates only expected tables and never rejects unexpected restored tables. An auditor-run control supplied one expected table plus one unexpected actual table; `_table_results` omitted the unexpected table and `RestoreDrillReport.ok` returned `True`. `civiccast/dr/restore_drill.py:126-145` captures only extension and sequence names. No code captures or compares constraints, sequence values/ownership, roles, object ownership, or grants. `civiccast/dr/backup.py:210-232` restores with `--no-owner`, and `create_fresh_postgres_database` creates only a new database on the same cluster, where roles already exist. PostgreSQL's own documentation states that `pg_dump` does not save cluster-global roles and that `pg_dumpall` is required for them: <https://www.postgresql.org/docs/17/app-pg-dumpall.html>.
- **Consequence:** the drill can report success although the recovered database has an extra table, lost/changed constraints, unsafe sequence state, missing ownership, or unusable grants/roles. That directly undermines the DR, cold-standby, and cutover database claims this gate is meant to close.
- **Acceptance criterion:** capture and compare both table-name sets; normalized application constraints; sequence definitions, current values, ownership, and grants; extensions; relevant roles/memberships; database/schema/table/sequence ownership and privileges; and table data. Prove globals against an independently fresh Postgres cluster or equivalent isolated destination rather than a second database that inherits the source cluster's roles. Add exact negative controls for at least an unexpected table, removed/changed constraint, altered sequence state, and revoked role/grant; each must make the real report fail. Re-run them under the exact-head CI floor.
- **Blast radius:** disaster recovery, cold standby, Postgres cutover, application authorization, referential/exclusion integrity, and post-restore writes.

### CC-WS2-002 - Major - CI proves helper composition, not the shipping Postgres drill entry point

- **Dimension:** tests / runtime / operability
- **Evidence:** `tests/dr/test_postgres_restore.py:147-173` directly calls `run_postgres_backup`, manually creates `BackupManifest`, and calls `run_postgres_restore_drill` with injected `docker exec` prefixes. No Postgres test calls `run_full_backup`, `run_full_drill`, the CLI command, or `civiccast.installer.service.run_dr_drill`; repository search finds top-level execution only in SQLite tests. The production dispatcher has no injection path for the test's command prefixes.
- **Consequence:** CI run `29559102912` proves that the lower-level helpers work together in a testcontainer. It does not prove that the shipping WSL operator path locates and invokes `pg_dump`, `psql`, and `pg_restore`, generates the real manifest, dispatches Postgres, writes the report, and returns the correct operator exit/result.
- **Acceptance criterion:** add an exact-head CI test that invokes the shipping `run_full_drill` path, preferably through the CLI or installer-facing entry point, against real Postgres using the same client-discovery/execution posture as the WSL product. Assert the generated report and operator exit/result, and add a dedicated JUnit floor proving this top-level test executed rather than skipped.

### CC-WS2-003 - Major - The manifest baseline is not tied to the dump's consistent snapshot

- **Dimension:** correctness / runtime
- **Evidence:** `civiccast/dr/backup.py:392-397` completes `pg_dump` and only afterward opens a separate SQLAlchemy connection to call `snapshot_tables`. `snapshot_tables` reads tables sequentially using the connection's normal isolation at `civiccast/dr/backup.py:84-109`. The test repeats the same order at `tests/dr/test_postgres_restore.py:149-163`. PostgreSQL documents that `pg_dump` has its own consistent snapshot and provides `--snapshot` for synchronizing another session: <https://www.postgresql.org/docs/17/app-pgdump.html>.
- **Consequence:** ordinary writes between dump completion and manifest capture can make a valid restore report false mismatch. DDL in that window can also combine with the one-way table comparison to hide an extra restored table. The committed manifest therefore does not reliably describe the bytes in `database.pgdump` on an active station.
- **Acceptance criterion:** bind `pg_dump` and the manifest queries to one synchronized exported snapshot, or enforce and prove an explicit quiesced backup contract. Run table snapshots in one repeatable transaction. Add a concurrent-write/DDL negative control that would expose the current split-snapshot implementation and passes only when dump and manifest represent one database state.
