## Executive result

The slice has substantial working machinery. The committed 76-test falsification suite and the full policy suite pass in the detached checkout. GitHub run `29572896732` is an exact-head pull-request run: all three producers and the `claims-verifier` job succeeded, the six downloaded artifacts carry run `29572896732`, attempt `1`, and source SHA `8b80967e...`, and the verifier emitted `claims-evidence: PASS`. That live pass nevertheless certifies the motivating false claim. The registered Postgres "backup/restore" claim is bound to `test_postgres_backup_snapshot_matches_source`, which calls `pg_dump` but never calls `pg_restore`; this tree does not contain `tests/dr/test_postgres_restore.py` or a Postgres restore implementation. The actual WS2 restore branch is not an ancestor of the audited SHA. The machine therefore passed exactly the kind of false "implemented and executed" statement gate 3 exists to reject.

Independent expected-red mutations also found three verifier-level false-pass classes: external claims with no non-CI control pass even while `owner_accepted: false`; typed roles need not identify the files their `where` and pytest node fields name; and misnamed/cross-wired producer artifacts plus a wrong `job_id` are accepted. The first-registration inventory also omits three program claims explicitly required by D9. These are functional proof defects under the owner severity decision, so the verdict is `CHANGES_REQUIRED`.

## Findings

### CC-WS3-001 - Critical - The live verifier certifies a Postgres restore claim although this SHA only executes `pg_dump`

- **Dimension:** correctness / false capability claim / required verification
- **Registry evidence:** `docs/claims/claims.yaml:57-107` states that the Postgres `pg_dump`/`pg_restore` path is implemented and executed against live Postgres, then binds the claim and its only control to `tests/dr/test_postgres_backup.py::test_postgres_backup_snapshot_matches_source`.
- **Bound-test evidence:** `tests/dr/test_postgres_backup.py:59-108` migrates and seeds one Postgres container, snapshots it, calls `run_postgres_backup`, and asserts only that the dump exists and is nonempty. The module imports `run_postgres_backup` and `snapshot_tables`; it never imports or calls a restore function or `pg_restore`.
- **Tree evidence:** the audited tree has no `tests/dr/test_postgres_restore.py`; `civiccast/dr/restore_drill.py` has no Postgres restore dispatcher; and `git merge-base --is-ancestor 7ed1a632... HEAD` is false. The separately audited WS2 implementation is not part of this SHA.
- **Live evidence:** downloaded `test-junit/junit.xml` records `tests.dr.test_postgres_backup::test_postgres_backup_snapshot_matches_source` as passed. It contains no Postgres restore test. The claims job then emitted `claims-evidence: PASS`.
- **Consequence:** gate 3's checker accepts a false "implemented and executed" claim at the exact head. This is the motivating overclaim the charter requires the checker to reject, not a wording issue.
- **Smallest acceptance criterion:** either remove/soften the Postgres restore claim until an audited WS2 merge is actually in the WS3 ancestry, or rebase WS3 on an exact WS2 `AUDIT_PASS` integration commit and bind this claim to a real end-to-end test that drives the shipping Postgres `run_full_drill` dispatch through dump, restore, and verification. Add a registry-level expected-red control proving that substituting a backup-only node cannot satisfy a backup/restore claim.
- **Blast radius:** charter gate 3, all registered capability claims, WS2/WS3 sequencing, future generated capability surfaces, and owner trust in a green claims job.

### CC-WS3-002 - Critical - External-evidence mode can return PASS while the trust root is explicitly non-authorizing

- **Dimension:** trust boundary / fail-closed behavior
- **Code evidence:** `_parse_entry` accepts an empty `controls` list at `scripts/policy/check_claims_evidence.py:318-320`. `run_external_evidence_mode` at `:1098-1103` skips every `ci_safe: true` control and performs no claim-level completeness check. The `owner_accepted` rejection exists only inside `resolve_external_evidence` at `:909-913`, so it is never reached when an external claim has zero controls or only CI-safe controls.
- **Executed falsification:** the auditor invoked the real CLI entry point twice with an `external_evidence` registry entry, the committed `owner_accepted: false` trust root, and deliberately missing bound files. With `controls: []`, output was `claims-evidence: PASS`, exit 0. With one `ci_safe: true` control, output was again `claims-evidence: PASS`, exit 0.
- **Additional fail-closed gap:** unlike same-run mode, `run_external_evidence_mode` does not execute `scan_marker_violations` or `blob_drift_violations`; once the trust root is accepted, external authority can resolve without verifying D1/D2 at the product commit.
- **Consequence:** a future external hardware/session-0 claim can become green with no external evidence and no owner acceptance merely by carrying no non-CI control. Even a non-vacuous external claim can ignore current product-side blob and marker drift.
- **Smallest acceptance criterion:** reject external entries unless they have at least one required `ci_safe: false` control; reject incompatible resolution/control combinations; check `owner_accepted` before iterating claims; require every external claim and control to resolve exactly once; and run D1/D2 checks in external mode. Add end-to-end CLI expected-red tests for empty controls, CI-safe-only external controls, unaccepted trust root, unmarked prose, and drifted bound inputs.
- **Blast radius:** external hardware, session-0, pristine-machine, and LPM evidence; audit-control authority records; owner trust-root acceptance.

### CC-WS3-003 - Major - Typed input roles are not tied to the identities they claim to bind

- **Dimension:** data provenance / schema enforcement
- **Code evidence:** `_parse_entry` parses `where.file`, `inputs.prose.path`, `inputs.test.path`, and `inputs.test.node_id` independently at `scripts/policy/check_claims_evidence.py:290-316`. It never requires the prose role to equal `where.file`, nor the path portion of the pytest node to equal the test-role path.
- **Executed falsification:** the real parser accepted an entry whose `where.file` was `actual-claim.md` while its prose role bound `unrelated.md`, and whose test role bound `unrelated_test.py` while its node named `tests/real_test.py::test_real`.
- **Consequence:** a registry can claim drift protection while binding unrelated bytes; subsequent changes to the real prose or real test can escape D2 without changing the recorded role blob.
- **Smallest acceptance criterion:** enforce `inputs.prose.path == where.file`; normalize and enforce the pytest node's file component equals `inputs.test.path`; reject duplicate or aliasing role paths where that would weaken identity; and add expected-red parser/CLI cases for both mismatches.
- **Blast radius:** every same-run and external claim, re-binding reviews, and registry generation.

### CC-WS3-004 - Major - Producer artifacts and metadata are not resolved by exact producer identity

- **Dimension:** CI provenance / artifact routing
- **Code evidence:** `find_producer_artifact` at `scripts/policy/check_claims_evidence.py:604-610` falls back from the contract's exact artifact directory to the first repository-wide filename match, ignoring `artifact_name`. Metadata checks at `:648-665` verify SHA and run ID but never verify `meta.job_id == producer`; `run_attempt` is emitted and documented by `emit_producer_meta.py` but is never checked.
- **Executed falsification:** the production function accepted a `junit.xml` stored under `wrong-junit-name`, a `test-meta.json` stored under `wrong-meta-name`, and metadata declaring `job_id: engine` while evaluating producer `test`. It returned no violations and populated test evidence.
- **Current-run boundary:** the six artifacts downloaded from run `29572896732` are correctly named and their metadata is correct. This finding is that the verifier does not enforce the exact mapping it and the workflow contract claim to enforce.
- **Consequence:** cross-wired, ambiguously downloaded, or stale-attempt artifacts can be attributed to the wrong producer and satisfy claim nodes or floors.
- **Smallest acceptance criterion:** resolve only the contract's exact artifact directory/name, reject fallbacks and ambiguous duplicates, require `job_id == producer`, and bind both `run_id` and `run_attempt` to the verifier's current context. Add expected-red tests for a wrong artifact directory, wrong job ID, duplicate filename, and prior-attempt metadata.
- **Blast radius:** all same-run claims, matrix/shard expansion, rerun attempts, and artifact download configuration.

### CC-WS3-005 - Major - D9 first registration omits release-truth, decision-gate, and session-0 program claims

- **Dimension:** scope / charter-contract coverage
- **Spec evidence:** `.agent-runs/native-windows/specs/spec-claims-evidence-rule.md:246-250` requires first registration of the program's WS2 drill, release-truth checker, decision gate, and session-0 claims, plus every strong token in `civiccast/dr/__init__.py` and the DR verification document.
- **Registry evidence:** `docs/claims/claims.yaml` contains only `ws2-postgres-restore-drill` and `ws2-sqlite-restore-falsification`. There is no entry for release truth, the media-worker decision gate, or the session-0 proof.
- **Scoping ruling:** deferring the broad README/CAPABILITIES/old-release sweep is acceptable under D9's "not a repo-wide sweep; the RULE ships first" sentence. It does not waive the four specifically enumerated native-Windows program claims. Those are a separate explicit first-registration obligation.
- **Consequence:** three of the program's highest-impact completion claims remain outside the rule at the gate intended to establish that rule.
- **Smallest acceptance criterion:** register the release-truth, decision-gate, and session-0 claims with exact current confidence boundaries, typed blob bindings, real tests or external controls as appropriate, and owner-nonauthorizing handling for external evidence until the trust root is accepted. Do not broaden this acceptance criterion into the deferred repo-wide sweep.
- **Blast radius:** STATUS/charter completion claims, future WS4/WS5 work that relies on the decision and session-0 proofs, and generated capability reporting.
