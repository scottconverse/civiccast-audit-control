## Findings

### CC-WS3-006 - Critical - External authority resolves unvalidated or non-JSON evidence as valid proof

- **Dimension:** trust boundary / proof-schema validation / false pass
- **Contract:** Charter section 8 requires commit-bound proof artifacts with a validated schema containing source SHA, command, environment/tool versions, exit status, timestamps, hashes, and provenance, and says an unvalidated proof file is `file_exists()` with extra steps. The passed v13 spec D5 repeats the full record body at lines 143-149, adding input blob IDs and origin machine.
- **Code evidence:** `resolve_external_evidence` at `scripts/policy/check_claims_evidence.py:1177-1194` catches `JSONDecodeError` by setting the body to `None`, then continues. If the body is a mapping, `source_sha`, `claim_id`, and `control_id` are checked only when present. No required evidence-record schema, command, environment, tool versions, timestamps, exit status, hashes, input blobs, origin, or raw-observation fields are validated.
- **Committed-suite evidence:** `test_d5_positive_resolution_succeeds` at `tests/policy/test_claims_evidence.py:1177-1189` treats `{\"v\": 1}` as a valid evidence record and passes.
- **Executed falsification:** the auditor created a local audit-control fixture using the production resolver and normal canonical-path/create-only/blob/authority checks, wrote the evidence bytes `THIS IS NOT JSON`, committed them, and supplied the fixture's fake-verifier seam as an auditor signature. `resolve_external_evidence(...)` returned `ok=True`, detail `resolved`.
- **Consequence:** after owner acceptance, a correctly placed and signed authority record can make arbitrary bytes satisfy a non-CI hardware/session-0 control. Signature and blob integrity authenticate the wrong thing; they do not supply the missing semantic validation.
- **Smallest acceptance criterion:** define and enforce an exact typed evidence-record schema for D5. Reject non-JSON, non-object, missing, extra where prohibited, or incorrectly typed fields. Require and validate at least the charter/spec fields: exact source/claim/control identity, exact command, environment and tool versions, timestamps, exit status/result, output hashes, bound input blob IDs, origin machine/provenance, and retained raw-observation references. Add independent expected-red tests for invalid JSON and every missing/invalid required field; replace the `{\"v\": 1}` positive with a complete valid record.
- **Blast radius:** all external hardware, session-0, clean-machine, and LPM claims; authority-record trust; owner acceptance of the trust root.

### CC-WS3-002 - Major - `PENDING_OWNER_ONLY=true` is emitted even when structural violations are present

- **Dimension:** fail-closed classification / CI outcome routing
- **Code evidence:** `run_external_evidence_mode` collects capability, marker, and blob violations at `scripts/policy/check_claims_evidence.py:1293-1296`. When `owner_accepted` is false, lines 1310-1316 append those violations to the error text but still raise `PendingOwnerAcceptanceError`. `main()` at lines 1399-1407 unconditionally converts that exception into `PENDING_OWNER_ONLY=true`.
- **Workflow evidence:** `.github/workflows/ci-test.yml:492-517` greps that marker and converts exit 2 plus `pending=true` into a successful warning specifically because pending owner acceptance is claimed to be the only cause.
- **Executed falsification:** the auditor ran the production CLI with one valid external entry, `owner_accepted:false`, and a post-binding mutation to its governed prose. The process found and printed structural violations including blob drift, exited 2, and emitted exactly `PENDING_OWNER_ONLY=true`. The audit-control placeholder remained untouched, so the intended owner short-circuit was also exercised.
- **Current-live boundary:** the preceding same-run step duplicates D1/D2 checks and was clean in run `29577106800`, so the exact live run did not conceal current drift. The external-mode machine-readable contract and warning conversion remain unsound on their own and can become the deciding path as wiring evolves.
- **Smallest acceptance criterion:** emit `PENDING_OWNER_ONLY=true` only when the pending-owner condition is literally the sole issue. If structural violations already exist, return them as a normal violation outcome or emit `PENDING_OWNER_ONLY=false`; the workflow must remain red. Add one end-to-end CLI/workflow-routing expected-red test combining `owner_accepted:false` with marker or blob drift.
- **Blast radius:** external-mode CI annotations, future owner-acceptance transition, and any caller using the marker as an outcome API.

### CC-WS3-005 - Major - External D9 registrations omit claim-defining modules and exact decision-gate configuration

- **Dimension:** D2 input completeness / claim-to-evidence identity
- **Contract:** v13 D2 says the `code` role carries the claim-defining module(s) and claim-specific fixtures/config are exhaustively listed. Blob drift only protects files actually enumerated.
- **Registry evidence:** the decision-gate evidence says it exercised `worker.py + engine.py + graph.py`; the session-0 evidence says it copied `{worker,engine,graph,control}.py`. Both registry entries bind only `civiccast/egress/gst/worker.py` as `inputs.code` (`docs/claims/claims.yaml:290` and `:362`). The schema/parser permit only one `code` file, while `blob_drift_violations` iterates only registered role files.
- **Configuration evidence:** the decision-gate evidence says no graph JSON from that run was committed and that its graphs came from `graph.demo_test_graph()`. Its registry entry instead binds `.agent-runs/native-windows/spike-session0/evidence/demo-graph.json` (`claims.yaml:314`), an equivalent later spike input rather than the exact configurations that generated the decision-gate observations.
- **Current-tree boundary:** all four engine files at the audited SHA are byte-identical to commit `3b4b2451...`; this is not a claim that current bytes differ. The defect is that future/current proof invalidation cannot detect drift in three expressly claim-defining modules, and the decision-gate config identity is substituted rather than exact.
- **Consequence:** a later source SHA can obtain external authority while changes to `engine.py`, `graph.py`, or `control.py` escape D2's product-side drift check. The decision-gate registration also presents a different graph as though it were the bound input to the observed run.
- **Smallest acceptance criterion:** make the code role one-or-more (or otherwise type every claim-defining module explicitly) and bind all modules named by each claim. For the decision gate, retain and bind the exact run configurations/raw input identities, or narrow the registered claim/control so it does not assert exact binding to unavailable inputs. Add omission mutations proving each additional module/config independently invalidates the claim.
- **Blast radius:** decision-gate and session-0 claim freshness, D2 re-proof/re-bind behavior, future supervisor/runtime work that relies on those claims.
