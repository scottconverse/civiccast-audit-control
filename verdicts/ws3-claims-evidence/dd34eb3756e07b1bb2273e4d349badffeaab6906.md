## Findings

### CC-WS3-006 - Critical - Evidence body accepts unrelated input blobs as proof for the registered claim

- **Dimension:** trust boundary / evidence-to-source identity / false pass
- **Contract:** Charter section 8 requires commit-bound proof with a validated schema covering environment/tool versions, hashes, and provenance. The passed v13 D5 contract and the round-2 acceptance criterion expressly require bound input blob IDs, not merely SHA-shaped strings.
- **Code evidence:** `validate_evidence_record_body` at `scripts/policy/check_claims_evidence.py:1119-1265` receives only `source_sha`, `claim_id`, and `control_id`. Lines 1244-1253 check that `input_blob_ids` is a non-empty list of 40-hex strings, but cannot compare it to `claim.role_files()`. `resolve_external_evidence` calls it at line 1444 without passing the claim's registered inputs. Lines 1190-1195 accept any non-empty mapping as `environment`; no tool name or version is required.
- **Committed-suite evidence:** the nominal body helper at `tests/policy/test_claims_evidence.py:632-647` uses a hard-coded `bbbb...` input ID. No test changes that value to a different valid 40-hex blob while holding authentication and the claim registry valid.
- **Executed production-resolver falsification:** the auditor built a temporary audit-control history satisfying canonical path, create-only history, evidence blob identity, authority identity, auditor-role signature, and no-retroactive-authority checks. The claim's code input was blob `cccccccccccccccccccccccccccccccccccccccc`; the evidence body named `bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb`, used `environment={"hostname":"machine-with-no-tool-version"}`, and referenced a nonexistent-looking raw path. `resolve_external_evidence(...)` returned `ok=True`, detail `resolved`.
- **Independent direct control:** the production validator accepted the same unrelated 40-hex input ID and hostname-only environment and printed `SCHEMA_ACCEPTED_UNBOUND_BODY`.
- **Consequence:** once owner acceptance enables external mode, an auditor-signed authority record can authenticate evidence produced from different code/config bytes and resolve the current claim. Authentication and immutable history preserve the wrong semantic association.
- **Smallest acceptance criterion:** bind the evidence body to the registered claim inputs. Pass the claim's role-file identities into validation and require an exact, deterministic match, preferably structured as role/path/blob entries; if retaining a flat list, compare the full multiset against every registered role-file blob and define duplicate handling. Require a machine-validated tool/version structure rather than any mapping. Add a production-resolver expected-red with all authorization checks valid but one different, well-formed 40-hex input blob, plus an environment-without-tool-version expected-red. The positive must derive its input identities from the claim fixture rather than an unrelated constant.
- **Blast radius:** every external hardware, session-0, clean-machine, and LPM claim after trust-root acceptance; authority-record semantics; future source-SHA reauthorization.

### CC-WS3-007 - Minor - Declared schema and review surfaces are not yet reconciled to runtime/current head

- `docs/claims/claims-schema.json` declares `additionalProperties:false` for registry structures, while `_parse_role_file`, `_parse_control`, and `_parse_entry` do not reject unknown keys. The auditor directly passed an unexpected role-file key through `_parse_role_file` successfully. Either enforce the declared contract in the runtime parser or label the schema as documentation that is not parser-authoritative.
- PR #293's live body still describes the round-1 state: 76 tests, two claims, and the superseded `ws2-postgres-restore-drill` claim. The audited tree/live run has 168 claims tests and five registered claims.
- The evidence README uses appended round narratives rather than a single current top-level summary, increasing the chance that reviewers consume superseded counts.
- **Smallest acceptance criterion:** reconcile parser/schema behavior and rewrite the PR/evidence top-level summaries to the current head. This item is batched and non-blocking under the owner's prose-precision calibration.
