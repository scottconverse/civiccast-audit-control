## Findings

### CC-WS3-008 - Major - Relevant restore marker can be removed while the governed surface still reports clean

- **Dimension:** claim-to-evidence association / marker adjacency / functional false pass
- **Contract:** D1 requires a governed strong claim to reference its registry entry with an adjacent marker. The round-4 smallest acceptance criterion required the scanner to reject the Postgres backup/restore block when its distinct restore association is absent.
- **Final-tree evidence:** `civiccast/dr/__init__.py:23` carries the new `ws2-postgres-restore-drill` marker. The same Python bullet later carries `ws2-postgres-backup-capture` at line 53 and the unrelated `ws2-sqlite-restore-falsification` marker at line 60. The SQLite entry's claim text contains `restore`.
- **Executed real-registry falsification:** copied every enforced governed file and every entry anchor target, removed only `(# claim:ws2-postgres-restore-drill)` from the exact real module text, left the capture and SQLite markers unchanged, loaded the real registry, and called `scan_marker_violations`. Result: `[]`.
- **Test blind spot:** `tests/policy/test_claims_evidence.py:2836-2838` removes both the new Postgres marker and the unrelated SQLite marker before asserting red. Its docstring acknowledges that leaving the SQLite marker would pass on an unrelated word choice. The test proves capture-only lexical coverage, but not loss of the actual relevant association in the real final block.
- **Independent Markdown falsification:** `scripts/policy/check_claims_evidence.py:781` assigns `sub_markers = marker_re.findall(sub_block) or markers`. A Markdown paragraph with an unmarked `Postgres restore is implemented` bullet and a marker only in a sibling bullet returned `[]`, because the empty local marker set fell back to the enclosing paragraph's marker set.
- **Consequence:** a future deletion or displacement of a capability's actual marker can leave `claims-evidence: PASS` when another entry in the same Python bullet or Markdown paragraph happens to use the same generic capability token. The registry may still bind excellent code and tests, but the governed claim is no longer tied to that entry.
- **Smallest acceptance criterion:** add an expected-red regression that removes only `ws2-postgres-restore-drill` from the unchanged real final block while leaving every other marker intact, and require a `restore` association violation. Add the sibling-Markdown case where the only covering marker is in another bullet. Change adjacency/coverage logic or the governed block structure so both cases fail, while a marker in the strong-claim bullet passes.
- **Blast radius:** all multi-marker Python bullets; all enforced Markdown paragraphs containing sibling bullets; D1 integrity for capabilities sharing generic lexicon words such as `restore`, `backup`, or `recovery`.
