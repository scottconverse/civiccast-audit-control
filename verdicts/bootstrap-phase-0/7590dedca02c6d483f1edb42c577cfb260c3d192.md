## Finding

### CC-BP0-001 remains open — Critical — A clean fresh Windows audit-control checkout false-blocks a valid pass record

**Runtime surface and code:** `scripts/policy/check_slice_audit_pass.py:197-213` reads the committed blob as bytes, reads the working file as bytes, and requires raw equality. The audit-control repository has no `.gitattributes` rule for verdict Markdown, while this machine's standard Git installation sets `core.autocrlf=true` in `C:/Program Files/Git/etc/gitconfig`. A fresh checkout therefore stores CRLF bytes in the working file while Git correctly retains LF in the blob.

**Evidence:** a real audit-control clone with canonical GitHub origin was given a hypothetical structured `AUDIT_PASS` record in the exact canonical format, including the audited SHA and this thread ID. The locally written LF record first passed the hook with exit 0. The repository was then freshly recloned under the standard Windows Git configuration and its origin set to the same canonical GitHub URL. The reclone was clean and the authority commit remained a 40-hex commit object, but:

- committed blob hash: `94c0f7683a431c5cc1e325ff11806b5c23a6bcd7`
- raw working-file hash (`git hash-object --no-filters`): `a8f0265b71326851c2723b086e9847984e4e07ba`
- Git-filtered working-file hash (`git hash-object`): `94c0f7683a431c5cc1e325ff11806b5c23a6bcd7`
- `git status --porcelain=v1`: empty
- documentation-exact TaskCompleted replay: exit 2, `working bytes differ from the blob at audit_control_commit`

The same raw-versus-filtered mismatch is present in existing files freshly cloned from the live audit-control repository, including the prior canonical `e45b92d9...` verdict. The committed tests write the canonical fixture after repository creation with explicit LF endings, so they do not exercise checkout smudge behavior.

**Consequence:** on the ordinary Windows configuration used by this native-Windows program, an authentic, tracked, clean, structured canonical pass record can be rejected solely because of Git's transparent line-ending conversion. The completion gate cannot reliably consume its canonical authority record.

**Blast radius:** every native-Windows slice using this local audit-control checkout path can false-block after a fresh clone or checkout. The failure is limited to advancement availability; the forged-authority mutations now fail closed as intended.

**Acceptance criterion:** make the working-copy integrity check Git-filter-aware while retaining tamper detection and binary-safe parsing, or enforce and prove raw LF working bytes for canonical verdict files through governance attributes and checkout handling. Add a regression that commits an LF canonical pass record, obtains it through a fresh checkout with `core.autocrlf=true`, proves the clean valid path exits 0, then modifies the record and proves the tampered path exits 2.
