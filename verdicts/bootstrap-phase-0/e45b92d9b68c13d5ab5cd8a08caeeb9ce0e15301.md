## Findings

### CC-BP0-001 — Critical — Canonical verdict validation can authorize a non-pass record

**Evidence:** `scripts/policy/check_slice_audit_pass.py:145-166` treats `audit_control_commit` as an arbitrary Git revision expression, never checks `verdict_repo`, accepts the canonical record when the substring `AUDIT_PASS` appears anywhere in its text, and compares decoded text rather than bytes. Three independent expected-red tests all exited 0:

1. canonical content declared `verdict: CHANGES_REQUIRED` but its rationale said `AUDIT_PASS is not warranted`;
2. cache declared `verdict_repo=https://example.invalid/forged`;
3. cache declared `audit_control_commit=HEAD` rather than a 40-hex governance commit SHA.

The command `python -m pytest bootstrap_phase_0_falsifications.py -q` returned `3 failed`; each failure reported `expected ... to block, got 0`. The harness used a documentation-exact TaskCompleted payload and a real temporary Git repository. This falsifies the code comment that all authority conditions hold and the test module's claim that it exercises the full section 8 validation chain.

A fourth control used the actual clean audit-control checkout and the newly committed canonical record at `ae9447939454a2b01df65163218b52959e7c6df3`. It exited 2 with “differs from its content at audit_control_commit” even though `git status` was clean and `git hash-object` exactly matched the committed blob. Diagnosis: `_git(..., text=True)` decoded Git's UTF-8 blob through the Windows locale, turning the first em dash into mojibake, while `Path.read_text(encoding="utf-8")` decoded the working copy correctly. The lengths differed (`11486` versus `11474`) at the first non-ASCII character. Thus the claimed byte-identical valid path also rejects a clean canonical record containing normal UTF-8 prose.

**Consequence:** after the encoding mismatch is removed, a canonical `CHANGES_REQUIRED` verdict can be mistaken for authorization merely because explanatory prose contains the pass token. Before that, a legitimate UTF-8 canonical record can fail closed despite being tracked, clean, and blob-identical. The coder-writable cache can also name a different repository and a moving Git ref. The completion hook therefore neither reliably accepts the valid authority path nor reliably rejects non-pass authority records.

**Blast radius:**

- Adjacent code/docs: `tests/policy/test_slice_audit_pass_hook.py`, `.agent-runs/native-windows/README.md`, `CLAUDE.md`, and `CHANGELOG.md` all describe this hook as canonical authority.
- User-facing/program effect: every native-Windows slice completion that relies on this hook can be marked complete without an exact canonical pass verdict.
- Migration: existing local caches must use a strict canonical URL and 40-hex governance commit after the fix; no persisted product data changes.
- Tests to add: exact canonical `CHANGES_REQUIRED` token rejection even when prose mentions the pass token; wrong `verdict_repo`; non-hex/moving governance ref; checkout-origin mismatch; exact source/thread field parsing.

**Acceptance criterion:** compare canonical blob/working-copy bytes without locale decoding; parse and compare exact structured canonical fields rather than substrings; require the canonical verdict field to equal the pass token, the source SHA and thread ID fields to equal the cache/current request, `verdict_repo` and checkout origin to identify `scottconverse/civiccast-audit-control`, and `audit_control_commit` to be a 40-hex commit object. All forged mutations above must exit 2; a tracked, clean, UTF-8 canonical pass record containing non-ASCII prose must exit 0.
