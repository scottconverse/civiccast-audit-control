## Executive result

`CHANGES_REQUIRED` is returned for exactly `aae0102786f2c96f726a37314cc3b96c5f90c475`.

The re-audit independently confirms that CC-WS5-001, CC-WS5-002, CC-WS5-004,
CC-WS5-005, CC-WS5-006, and CC-WS5-011 are materially remediated. The boot-held
and ready-to-held paths now preserve process identity and replace a normal
control plane with one maintenance-mode process; repeated unattested ticks do
not duplicate it. The continuous serving guard consumes the real raw
`action="refuse", state_name=None` shape without crashing and controlled-stops

CC-WS5-003 still assigns the short-lived `pg_ctl start` launcher PID rather than

the WS5 build basis; only the live Session-0/real-Postgres validation venue is
owner/VM-dependent. The missing implementation remains coder work.

Fresh execution also broadens F-REV-6. It is not limited to a very-low-probability
clear-to-refuse flip between two adjacent evaluations. When WSL is already
authoritative as maintenance is released, the real `GuardMonitor.evaluate_once()`
returns `action="never_start", state_name=None`; the maintenance-release router
misclassifies `state_name=None` as clear, terminates the maintenance control

is left `stopped`; later ticks never evaluate or retry it, even after the guard
clears. This is fail-closed, so it is Major rather than Critical, but it is a
deterministic indefinite service-availability defect for that input and is
coder-actionable.

## Closed prior findings

### CC-WS5-001 - Closed - Maintenance process identity and spawn idempotence

`Supervisor.start()` now polls the interlock before the writer-capable child;
`_cp_mode` distinguishes normal and maintenance children; and
`_reconcile_maintenance()` terminates a normal control-plane PID before starting
one maintenance PID. Independent tests asserted the environment mode, distinct
old/new PIDs, termination of the normal PID, and exactly one live maintenance
spawn across repeated unattested ticks.

### CC-WS5-002 - Closed - Continuous dual-runtime guard controls the writer

`tick()` now evaluates the guard at `guard_interval_seconds` while serving,
routes all non-start actions fail-closed, and controlled-stops the control plane.
An independent composition using the real `GuardMonitor` produced raw
`refuse/None`; the supervisor entered `blocked_wsl_active`, removed the writer
handle, and terminated the original PID. Routing the mid-operation decision on
`action`, rather than the design prose's `state_name`, is the correct safety
implementation for the actual consumed API shape.

### CC-WS5-004 - Closed - RAT-004 terminal command precedes escalation

The drain owner sends the worker's terminal `stop` command when the strategy
supports it, observes process exit as ground truth, and terminates only after the
deadline. Focused tests cover no premature terminate, lost acknowledgement with
process exit, and hung-worker escalation.

### CC-WS5-005 - Closed - Recovery, jittered backoff, downstream restart, and recovered event

Focused tests prove termination/replacement of a live-unready child, deterministic
per-child jitter epochs and reset, D6-ordered downstream restart, and a single
`recovered` dispatch after fresh readiness.

### CC-WS5-006 - Closed - D2 correlation and production lifecycle

Focused tests prove waiter registration before transport write, explicit lost-ack
logging, reload/swap-only replay, same-channel replacement, daemon reconnect,
and close on stop/shutdown.

### CC-WS5-011 - Closed - Diff hygiene

`git diff --check 0d042dd2f6bd3873608c4d257e4e9f1455c910ab..aae0102786f2c96f726a37314cc3b96c5f90c475`

## Open findings

### CC-WS5-003 - Critical - Durable PostgreSQL ownership is still not implemented

`civiccast/native/supervisor/children.py:224` still launches
`pg_ctl start -D ... -w`. `Supervisor.start_child()` assigns the returned
launcher handle at `civiccast/native/supervisor/core.py:452-461`. No production
code reads or validates `postmaster.pid`, opens the durable postmaster process,
or replaces the tracked launcher handle before readiness is accepted.

The auditor re-ran the realistic launcher falsification: the injected `pg_ctl`
launcher exited immediately while the PostgreSQL readiness probe succeeded.
Startup reached `ready`; three ticks produced four PostgreSQL launches plus
ordered dependent restarts, and every assigned PID was a spawned launcher or
dependent rather than a discovered durable postmaster. The source still cannot
prove AC2/AC3/AC4 containment or monitor the actual database server.

**Scope disposition:** reading and validating `postmaster.pid`, opening the real
process, assigning it to the Job Object, and monitoring that handle are
coder-buildable and fake-testable. Real PostgreSQL/Session-0 containment proof
is VM-bound. The finding cannot be scoped wholesale to owner/VM work.

**Required acceptance:** implement and unit-falsify durable postmaster PID/handle
acquisition and observable stop failure, then execute the real PostgreSQL start,
unexpected death, restart, service stop, and supervisor-death containment panel
on the owner-provided Windows venue.

**Blast radius:** PostgreSQL restart storms, duplicate/contending postmasters,
WAL-safe shutdown, Job Object kill-on-close, and AC2/AC3/AC4 evidence.

### CC-WS5-012 - Major - Maintenance release deterministically wedges when the raw guard blocks

`_interlock_freed_event()` at `civiccast/native/supervisor/core.py:984-1002`
uses `decision.state_name`; `None` is classified as
`interlock_freed_clear`. Real guard decisions such as selector `wsl` return
`action="never_start", state_name=None`. The false-clear event moves maintenance
to `starting`; `_resume_to_serving()` terminates the maintenance control plane
and `start_child()` correctly withholds the normal writer, leaving the child
`stopped` at `core.py:760-779`. `_needs_restart()` then permanently excludes it
at `core.py:842-853`.

Independent execution with an actual `GuardMonitor` produced:

```text
after_release state=starting child_state=stopped cp_present=False guard=never_start/None
after_guard_clear state=starting child_state=stopped cp_present=False control_plane_spawn_count=1
```

This is broader than PENDING.md's disclosed within-one-tick clear-to-refuse race:
an already-blocked WSL-authoritative release reproduces it deterministically.
It remains fail-closed—no writer starts—so Major is appropriate, but the service
does not recover automatically after the guard clears.

**Required acceptance:** route maintenance-release authorization on the same
action vocabulary used by the continuous guard, or otherwise ensure a non-start
decision enters the correct blocked state; preserve one-evaluation-per-edge;
make a withheld normal control-plane start restartable after authorization
returns; and add actual `GuardMonitor` composition tests for `never_start/None`,
`refuse/None`, `blocked_probe_unavailable`, and clear.

**Blast radius:** RAT-002 maintenance exit, service availability after upgrades,
operator state truth, control-plane recovery, and the F-REV-6 disclosure.

### CC-WS5-009 - Minor - Whitespace-only explicit mode still fails open to normal

`civiccast/app.py:485-487` treats both `raw is None` and `raw.strip() == ""` as
backward-compatible normal. The function's contract says only a genuinely absent
environment value is normal and every explicitly present unrecognized value
fails closed. Independent execution with `CIVICCAST_SUPERVISOR_MODE=' '` returned
`normal`. Existing tests cover nonblank unknown strings but not blank/whitespace.

**Required acceptance:** distinguish absent from present-but-blank and add the
blank/whitespace negative controls.

### CC-WS5-010 - Minor - Status-snapshot disclosure understates the race

The finding remains open. In addition to a torn successful view,
`status_snapshot()` performs a membership test and later indexes the mutable
handle dictionary at `civiccast/native/supervisor/core.py:944-950`. A concurrent
handle removal between those operations can raise `KeyError`; the command queue
converts that into a transient `status:error` response. PENDING.md and the method
docstring currently claim no crash/corruption and only a momentarily torn view.

**Required acceptance:** implement a lock-free consistent copy/build snapshot or
equivalent nonblocking mechanism, test the removal interleaving, and correct the
disclosure until closed.
