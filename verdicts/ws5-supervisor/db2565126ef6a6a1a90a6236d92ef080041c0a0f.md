## Findings

### CC-WS5-007 — Critical — registered SCM service class is not import-resolvable

`civiccast/native/supervisor/service.py:872-932` defines
`CivicCastSupervisorService` as a function-local class. `main()` builds that
class and passes it to `win32serviceutil.HandleCommandLine` at
`service.py:1022-1045`. pywin32's `GetServiceClassString` persists the class as
module/file plus `cls.__name__`, not its function-local `__qualname__`.

The auditor reproduced both supported invocation shapes. Imported-module
construction produced:

```text
module=civiccast.native.supervisor.service
name=CivicCastSupervisorService
qualname=build_service_class.<locals>.CivicCastSupervisorService
registered=civiccast.native.supervisor.service.CivicCastSupervisorService
module_has=False
```

The documented `python -m civiccast.native.supervisor.service install` shape
likewise caused pywin32 to persist the service file plus
`.CivicCastSupervisorService`; importing the file in the later service-host
process still creates no module-global class. Existing Windows tests prove only
the in-process subclass shape and injected handler call. They do not resolve the
persisted class string from a fresh import.

The same cross-process boundary invalidates the proposed dependency injection.
`build_production_service_factory()` returns an installer-process closure, and
`default_dependency_provider()` at `service.py:964-981` unconditionally raises
`NotImplementedError`. A VM can prove a concrete production composition after
it exists; it cannot inject that closure into a separately started SCM host.

**Required acceptance:** expose a module-level, import-resolvable
`ServiceFramework` class whose `SvcDoRun` constructs the concrete production
dependencies in the service-host process; add a separate-process class-string
resolution falsification; then run the owner-VM SCM/session-0 proof.

**Blast radius:** the entire native supervisor product: Session-0 boot, child
containment/recovery, control pipe, maintenance, admin commands, and every live
acceptance criterion depend on the service host loading.

### CC-WS5-012 — Major — correct block routing still wedges after the guard clears

The action-vs-state routing defect is fixed, but liveness is not. `tick()` at
`civiccast/native/supervisor/core.py:736-761` evaluates the continuous guard only
in `ready` or `degraded`. Both `blocked_wsl_active` and
`blocked_probe_unavailable` fall through without guard evaluation. The state
machine already defines `blocked_* --guard_clear--> starting` at
`states.py:170-175`, but the supervisor never dispatches that event from a
blocked state. Admin start/restart is also deliberately refused while blocked.

An independent real-`GuardMonitor` composition entered maintenance, released to
`blocked_wsl_active`, changed the selector to a start-authorizing native result,
and ticked three times with the interlock continuously free. The final state was
still `blocked_wsl_active`, the control plane remained stopped, and the last
guard result was unchanged. The committed recovery test masks the gap by
supplying a second `held -> free` interlock cycle.

**Required acceptance:** reconcile the real guard while blocked, dispatch
`guard_clear` on a start-authorizing verdict, and prove both blocked states
recover with the interlock continuously free and without weakening the
one-evaluation-per-release-edge rule.

**Blast radius:** fail-closed station outage after a transient WSL/probe block;
recovery currently requires a service restart or another interlock cycle.

### CC-WS5-015 — Major — postmaster attachment faults escape and stop supervision

The expected postmaster path at `core.py:509-535` is materially correct under
the narrow fakes: the launcher is not assigned, the postmaster PID is read,
`open_existing(P)` is called, the postmaster handle is swapped and assigned,
three stable ticks do not relaunch PostgreSQL, missing PID data fails closed,
and postmaster death triggers ordered recovery/rebind.

However, `open_existing()` and `assign_child()` at `core.py:529-531` have no
exception boundary. Independent hostile seams showed `OSError` escaping for
both an OpenProcess failure and a Job-assignment failure. `SupervisorService.run`
has teardown in `finally` but no recovery boundary at `service.py:644-651`, so
the exception exits supervision. In the assignment-failure case the tracked
handle has already been replaced even though containment was not established.

**Required acceptance:** convert open/assignment faults to an explicit failed,
observable, retryable fail-closed outcome; keep handle/child/job state coherent;
and add tests for both faults before the live-postmaster VM panel.

**Blast radius:** service availability, PostgreSQL containment, restart storms,
and the integrity of AC2/AC3/AC4 evidence.

### CC-WS5-007-TOCTOU — Major — owner verification and send use different pipe handles

`civiccast/native/supervisor/control_client.py:195-224` opens handle A, reads
the server owner SID, and closes it. `_default_transport()` at lines 227-246
opens handle B and sends the command. An injected endpoint-replacement
falsification verified that a trusted owner can be observed on A while the
command is sent to an attacker-owned B.

The decision logic correctly refuses untrusted/unreadable owners and does not
call transport in those cases. The defect is endpoint binding across the two
production opens. Exploitation requires a narrow local service-death/recreation
race and primarily enables command disclosure/operator deception rather than
mutation of the legitimate service, so Major rather than Critical is assigned.

**Required acceptance:** open once, verify owner on that same live handle, and
send/read on that handle; then prove the same-handle path against a real named
pipe on the VM.

**Blast radius:** anti-squat trust, operator command confidentiality/integrity,
and all control-client verbs.

### CC-WS5-017 — Minor — admin `start` from `starting` is non-idempotent

`AdminCommandRouter._start()` falls through to `Supervisor.start()` when state
is `starting`. The real Supervisor with narrow fakes respawned PostgreSQL and
then spawned the remaining children, producing two PostgreSQL spawns and
overwriting tracked state. The path is admin-only and limited to the bring-up
window, so Minor is retained.

**Required acceptance:** treat `starting` as a no-op or make repeated startup
handle-aware and prove it cannot duplicate a live child.
